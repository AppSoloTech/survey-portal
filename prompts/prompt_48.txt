# Received Feedback

1. Admins need to review individual open-text answers in Results.
2. Admins need to apply existing catalog tags to a specific text answer.
3. Automatic hidden/value/Other tags should remain visible but read-only.

# Intake Classification

* Admin workflow feature
* reporting UI change
* no participant-facing behavior change
* depends on Phase 47 API contracts

# Scope

Implement:

* Fetch the existing tag catalog once on the Results page.
* In expanded attempt detail, render editable review-tag chips only for answered
  `text` responses that have a `responseAnswerId`.
* Add a compact catalog picker that excludes already-applied tags.
* Remove review tags with pill controls.
* Call Phase 47 add/remove API helpers and update local attempt detail state.
* Keep automatic hidden tags, value tags, and Other tags read-only.
* Add focused styles reusing the current Results/tag chip patterns.

## Non-Goals

Do not implement:

* Backend behavior from Phase 47 if it is not already present.
* Integer/manual numeric tagging.
* Other-text manual tagging.
* Free-form tag creation inside Results.
* Review-tag rollup charts.
* Participant-facing UI changes.

## Assumptions

* Catalog management remains in `/admin/tags`.
* Results tagging uses existing catalog tags only.
* Failed tag mutations should preserve current detail state and show an error.

# Acceptance Criteria

The phase is ready for review when:

* Admin can expand an attempt, add a catalog tag to a text answer, remove it,
  refresh, and see persistence.
* Tag controls do not appear for choice, scale, integer, skipped, blank, or
  never-reached answers.
* Already-applied tags are excluded from the picker.
* Automatic tags remain read-only and visually distinct.
* Results remains usable at 375/768/1280px.
