# Phase 29 — Tag Group Management Foundation

## Purpose

Add Admin-managed grouping for the tag catalog so category/value tag pairs can
be housed, ordered, and managed more easily on `/admin/tags`.

This phase depends on Phase 28 terminology alignment. Groups organize existing
tag category/value pairs as intact pairs; a category/value pair should never be
split across separate grouping primitives.

# Received Feedback

Implement a grouping feature for tag housing and management. Implementation
should be similar to the `/admin/surveys/2/organize` question grouping and
manipulation drag/drop experience. The `/admin/tags` page needs a section to
create new groups and should smartly handle new tag creation.

Clarified product decision:

* Groups organize tag category/value pairs together.
* Tag pairs should stay together as they do now.
* Deleting a group does not delete tag definitions; contained tags return to
  the public ungrouped holding area.

# Scope

Implement:

* Admin-only tag groups with a name, display order, timestamps, and delete
  behavior that does not delete tag definitions.
* Assignment of catalog tag definitions to zero or one group by default.
* Drag/drop ordering for groups and for tags within a group, modeled after the
  existing Organize tab patterns.
* An ungrouped section for catalog tags with no group.
* New tag creation on `/admin/tags` with optional group assignment.
* Editing a tag's category/value without losing its group assignment.
* Moving a tag pair between groups while keeping the category/value pair intact.
* Shared Admin response types for grouped catalog data.
* Focused tests for group CRUD, assignment, reorder, and admin-only access.

Use the next available migration number at implementation time. Do not hard-code
the migration number in this prompt or docs because earlier phases may change
the sequence.

## Non-Goals

Do not implement:

* Splitting categories and values into separate groupable objects.
* Allowing one tag pair to belong to multiple groups.
* Participant-facing tag group payloads.
* Changes to hidden-tag matching, reporting counts, CSV exports, response data,
  or survey validation.
* Cross-survey tag rollups or reporting filters.
* Grouping survey questions; this should only borrow interaction patterns from
  the Organize tab.

## Assumptions

* A group is admin-only catalog metadata.
* Existing tag definitions start ungrouped after migration.
* Deleting a group leaves its tags in the catalog as ungrouped.
* Existing hidden tags saved on answer options, Other, or value tags continue to
  use category/value pairs exactly as before.

# Acceptance Criteria

The phase is ready for review when:

* Admins can create, rename, reorder, and delete tag groups.
* Admins can assign existing category/value tag pairs to groups or leave them
  ungrouped.
* Admins can drag/drop groups and tag pairs within/between group sections.
* New catalog tag creation can optionally assign the tag pair to a group.
* Non-admin users cannot access tag group APIs or pages.
* Participant APIs, reports, hidden-tag rollups, CSV output, and survey
  response storage remain unchanged.
