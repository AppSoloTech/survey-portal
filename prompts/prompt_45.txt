# Phase 45 - User Dashboard And Public Directory Accessibility Polish

Remediate remaining public and registered-user accessibility findings from
`docs/ACCESSIBILITY_AUDIT_REPORT.md` after the survey runner, app shell/modal,
and form feedback phases have landed.

dashboard, category survey lists, public anonymous survey directory, survey
summary cards, user-facing repeated action names, any remaining visible status

# Received Feedback

User-facing surfaces need remaining polish after the critical phases:

1. Repeated action buttons such as "Start survey", "Resume survey", "View
   completed", and anonymous "Start survey" need contextual accessible names
   where surrounding card context is only visual.
2. Important disabled user actions that remain after Phase 44, such as
   disabled survey/contact actions, should explain why they are unavailable.
3. Inline glossary behavior should preserve the existing focus/hover/Escape
   interaction while avoiding unsupported or unreliable ARIA where practical.

Phase 44 baseline now in place:

* `AlertMessage` owns scoped public/user status semantics for error, success,
  and info messages.
* Auth and account forms already use `FormField` for required/optional markers,
  helper/error associations, `aria-invalid`, and `aria-describedby`.
* Dashboard/category/public directory loading, empty, and error states already
  use the shared status primitive where Phase 44 touched them.
* Shared `PaginationRow` already renders the visible page status as a polite,
  atomic `role="status"` live region.
* Shared `ToastProvider` already uses persistent polite/assertive live-region
  containers with dedicated dismiss buttons.
* Account settings password reset cooldown and invalid reset-link explanations
  were addressed in Phase 44 and passed manual testing. Do not redo that work
  unless a direct regression is discovered.

# Scope

Implement:

* Add item-specific accessible names or visually hidden context to repeated
  user-facing survey action controls:
  * dashboard survey cards
  * category survey cards
  * category group cards
  * resume nudge
  * anonymous public directory survey cards
  * homepage anonymous survey entry point if needed
* Regression-check shared pagination status on dashboard/category pages; only
  change `PaginationRow` if Phase 44's polite live region proves insufficient
  in the specific Phase 45 surfaces.
* Review public/user loading, empty, success, and error states after Phase 44;
  only apply `AlertMessage` where scoped route states remain inconsistent.
* Add or improve user-facing disabled-state explanations where users need to
  understand why an action cannot be taken and Phase 44 did not already address
  the control.
* Tighten inline glossary accessibility only for participant-facing question
  prompt usage:
  * preserve existing focus, hover, click/tap, and Escape behavior
  * prefer robust `aria-describedby` behavior over unsupported
    `aria-description`
  * ensure changes do not alter survey answers, glossary matching, hidden tags,
    reporting, or CSV exports
* Keep visual changes small and consistent with the existing design.

## Non-Goals

Do not implement:

* Admin custom grouped question select fixes.
* Admin Results table/chart alternatives.
* Admin users mobile table fixes.
* Admin disclosure or drag-and-drop fixes.
* Admin repeated action names or disabled-state explanations.
* Survey runner core answer-control work already owned by Phase 42.
* App shell/modal route work already owned by Phase 43.
* Auth/account form-field foundation already owned by Phase 44.
* Reworking `AlertMessage`, `FormField`, `ToastProvider`, or `PaginationRow`
  unless a Phase 45 regression check finds a specific remaining user-facing
  issue.
* Account settings password reset cooldown explanation already owned and
  manually verified in Phase 44.
* API, database, hidden-tag, reporting, or CSV behavior changes.
* Broad visual redesign.

## Assumptions

* Contextual accessible names can be added with `aria-label` or visually hidden
  text without changing visible labels.
* Inline glossary fixes should be narrow because Phase 41 already implemented
  focus, hover, click/tap, and Escape behavior.
* Dashboard/category/public directory changes can be implemented entirely in
  React and CSS.

## Open Questions

* If `aria-label` would hide useful visible button text from some assistive
  tech behavior, prefer visually hidden contextual text inside the control.

# Acceptance Criteria

The phase is ready for review when:

* Repeated user-facing survey actions are understandable in screen reader
  button/link lists.
* Dashboard and category pagination continue to announce politely through the
  shared Phase 44 `PaginationRow` behavior.
* Public/user loading, empty, error, and success states remain semantically
  consistent with Phase 44 primitives.
* Disabled user-facing actions that remain in Phase 45 scope include
  discoverable reasons.
* Participant inline glossary trigger behavior remains available by keyboard,
  pointer, touch, and Escape.
* Admin-only audit findings remain out of implementation scope.
* Existing public, authenticated, anonymous, and registered-user survey flows
  still work.
