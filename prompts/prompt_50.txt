# Phase 50 — CLI Performance Test Harness And Result Persistence

## Purpose

Build the command-line performance testing harness that seeds load-test data,
runs HTTP and optional direct-DB performance checks from the operator's machine,
and persists summarized results into the Phase 49 `performance_test_runs`
table.

The Admin app remains a viewer only. This phase does not add a browser control
for starting tests.

# Scope

Implement:

* Create a top-level `loadtest/` harness.
* Add `.env.loadtest.example` with placeholders only.
* Add root npm scripts:
  * `loadtest:seed`
  * `loadtest:run`
  * `loadtest:db`
  * `loadtest:teardown`
* Update `.gitignore` for:
  * `.env.loadtest`
  * `loadtest/.manifest.*.json`
  * `loadtest/reports/`
* Reuse the hosted-target safety pattern from `scripts/migrate-hosted.mjs`:
  * explicit `.env.loadtest` parsing
  * host guard
  * TLS for hosted PostgreSQL
  * confirmation before hosted writes
  * clear target printing
* Seed namespaced load-test data directly with SQL.
* Teardown load-test rows by manifest and/or marker in safe dependency order.
* Run k6 HTTP scenarios from the CLI when k6 is installed.
* Persist a `running` row to `performance_test_runs` at run start.
* Update the row to `completed`, `failed`, or `aborted` at finish.
* Store concise JSON summaries and markdown reports.
* Keep local full artifacts under `loadtest/reports/`.
* Keep Azure metric sampling optional. If Azure CLI or permissions are missing,
  persist the HTTP/DB summary and label Azure resource metrics as unavailable.
* Add a non-Azure PostgreSQL metrics sampler that runs over the same load-test
  database connection during a test. At minimum, sample:
  * `pg_stat_activity` for active/total connection counts and application-side
    connection pressure signals.
  * `pg_stat_database` for transaction and block read/hit deltas.
  * optionally `pg_stat_statements` only when the extension is installed and
    the configured user can read it.
* Use the SQL sampler output in the bottleneck classifier before falling back
  to `unknown` when Azure Monitor metrics are unavailable.

## Load-Test Data Requirements

Seed data should:

* Use a stable marker such as `LOADTEST <run_key>`.
* Use load-test email prefixes such as
  `loadtest+<run_key>-<n>@example.invalid`.
* Create enough surveys, users, pages, questions, answer options, hidden tags,
  and completed attempts to exercise reporting endpoints.
* Write created ids to `loadtest/.manifest.<run_key>.json`.
* Avoid real client data.

Write-heavy scenarios must avoid one shared user completing the same survey
repeatedly. Use one of:

* Preferred default: the anonymous survey flow, where each iteration calls
  `POST /api/anonymous-surveys/:token/start` to create a fresh anonymous
  attempt, saves answers through the anonymous answer/page-answer endpoints, and
  completes with `POST /api/anonymous-surveys/:token/complete`.
* Alternative for registered-user behavior only: many seeded user/survey pairs
  large enough that the run cannot exhaust them.

Do not rely on one shared registered user. Registered `POST /api/surveys/:id/start`
returns 409 after that user has already completed the survey.

Do not use anonymous `/register` as the default write-throughput mechanism.
That endpoint converts an already-completed anonymous attempt into a registered
user and is useful only for a separate conversion-load scenario.

Anonymous write-load setup must account for rate limiting:

* All `/api/anonymous-surveys/:token/*` public routes share
  `ANONYMOUS_SURVEY_RATE_LIMIT_MAX` over
  `ANONYMOUS_SURVEY_RATE_LIMIT_WINDOW_MS`.
* The default is 120 requests per 15 minutes, and a single complete anonymous
  attempt consumes multiple requests.
* Document the temporary App Service setting increase needed for an approved
  test window, and reset the values afterward.
* If a conversion scenario uses anonymous `/register`, it is also governed by
  `AUTH_REGISTER_RATE_LIMIT_MAX` over `AUTH_RATE_LIMIT_WINDOW_MS`.

## Scenario Requirements

HTTP scenarios:

* read-heavy Admin reporting:
  * `GET /api/surveys/:id`
  * `GET /api/surveys/:id/report`
  * `GET /api/surveys/:id/attempts`
  * `GET /api/surveys/:id/attempts/:attemptId`
  * `GET /api/surveys/:id/export.csv`
* write-heavy survey completion:
  * default to anonymous public survey start
  * save page/question answers with the anonymous attempt token
  * complete the anonymous attempt
  * optionally include a separate anonymous-to-registered conversion scenario
    only when registration load is explicitly desired
* mixed read/write scenario

Thresholds to summarize:

* p50 latency
* p95 latency
* p99 latency
* error rate
* HTTP 5xx count
* total requests
* failed requests
* peak requests per second
* max VUs

Required SQL metrics to summarize:

* active DB connections during each step/window
* total DB connections visible to the configured user during each step/window
* transaction deltas from `pg_stat_database`
* block read/hit deltas from `pg_stat_database`
* whether app-facing DB connections appear pinned around the application pool
  size while latency/error rate rises

Direct DB load:

* Optional but useful.
* Run from the CLI environment only.
* Use configurable pool size and concurrency.
* Replicate heavy structure/reporting queries closely enough to distinguish DB
  pressure from app-tier `pg` pool pressure.

## Non-Goals

Do not implement:

* Admin UI pages.
* Browser controls to run, stop, or schedule tests.
* App-Service-hosted load generation.
* New Azure Container Apps, Container Instances, Azure Load Testing, queues,
  functions, or Blob Storage.
* Required Azure metric sampling.
* Live-network automated tests.
* Storing huge raw k6 event streams in PostgreSQL.
* Any change to production app runtime behavior except reading the new report
  table created in Phase 49.

## Assumptions

* Operators install k6 separately.
* Operators run the CLI during an approved testing window.
* The CLI can connect to the hosted database using `.env.loadtest`.
* The Phase 49 migration has already been applied to the target database before
  result persistence is attempted. For hosted tests, run
  `npm run db:migrate:hosted` first.
* The Admin UI in Phase 51 will read summaries from PostgreSQL, not local files.
* Raw local artifacts are useful for operators but are not the app's source of
  truth.
* Phase 50 direct database updates to `performance_test_runs` must set
  `updated_at = now()` explicitly when completing or failing a run.

## Open Questions

* Should `loadtest:run` default to a smoke scenario unless `--profile` is
  provided? If unanswered, prefer the safer default.
* Should Azure Monitor metrics be included in V1? If unavailable, do not block
  result persistence.

# Acceptance Criteria

The phase is ready for review when:

* `.env.loadtest.example` documents all required load-test configuration with
  placeholders only.
* Load-test secrets and generated reports/manifests are ignored.
* Seed creates clearly namespaced fake data and writes a manifest.
* Teardown supports dry-run and deletes only namespaced/manifested load-test
  data.
* A smoke `loadtest:run` can persist a started row and complete or fail it with
  useful summary data.
* Completing or failing a persisted run updates `updated_at = now()` in the
  direct database update SQL.
* k6 output is summarized into `performance_test_runs`.
* Failed runs update the row to `failed` with useful error context.
* Write-heavy scenarios use the anonymous start/answer/complete flow by
  default and do not collide on one shared completed registered user.
* Operator docs explain the temporary anonymous rate-limit env increase needed
  for meaningful anonymous write-load tests, and the reset step afterward.
* The SQL metrics sampler records `pg_stat_activity` and `pg_stat_database`
  data when permissions allow it, and gracefully labels unavailable SQL metrics.
* The harness refuses unsafe hosted/local target confusion unless an explicit
  dev mode is used.
* No additional Azure resources are required.

* Local/dev seed with explicit dev flag.
* Local/dev teardown dry-run.
* 1 VU / short-duration HTTP smoke against a safe target.

  attempt token.
* SQL sampler smoke: collect at least one `pg_stat_activity` and
  `pg_stat_database` sample, or document the permission/availability failure.
* Confirm a `performance_test_runs` row is created and updated.
* Confirm completed/failed persistence updates the row's `updated_at`.
* Confirm local artifacts are written under `loadtest/reports/` and remain
  ignored.

Do not require live k6, live Azure CLI, or live hosted DB access in automated
tests.
