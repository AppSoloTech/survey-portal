## Purpose

Build a safe anonymous survey foundation with tokenized public access while
keeping anonymous takers separate from logged-in users.

This is a foundation phase. Implement only the minimum anonymous survey-taking
path needed for tokenized access and safe attempt ownership.

# Received Feedback

Develop anonymous survey functionality with a safe bifurcation between logged-in
users and anonymous survey takers who do not have, and should not need, an
account.

# Scope

Implement:

* Durable documentation update allowing anonymous survey completion, replacing
  the previous "users must register and login" decision where needed.
* Tokenized anonymous link model scoped to a published survey, with enabled or
  disabled status and optional expiration if the implementation can keep it
  simple.
* Anonymous attempt ownership that is separate from `users.id`.
* Public anonymous survey read/start/answer/complete path that reuses the
  existing survey runner logic where practical.
* Admin controls to create, copy, and disable anonymous links for published
  surveys.
* Reporting display that identifies anonymous attempts without inventing user
  accounts.

## Non-Goals

Do not implement:

* user account creation for anonymous takers
* anonymous access to `/api/my-surveys`, user dashboard, profile, admin routes,
  hidden tags, or auth-only APIs
* broad email invitation workflows beyond optional link copy
* assignment rules or recipient list management
* public survey discovery pages

## Assumptions

* Anonymous tokens must be high entropy and stored safely.
* Anonymous attempts should not set auth cookies.
* Anonymous participants receive the same participant-safe survey shape as
  logged-in users: no hidden tags or admin-only metadata.
* Completed anonymous attempts should be visible in admin reporting.

## Open Questions

* Should anonymous links expire by default, or remain active until disabled?
  If unanswered, prefer a nullable expiration with no default expiry.

---

# Acceptance Criteria

The phase is ready for review when:

* Admins can create and disable an anonymous link for a published survey.
* Anonymous visitors can open a valid token, complete the survey, and submit
  without logging in.
* Bad, disabled, expired, or survey-unavailable tokens show safe unavailable
  responses.
* Anonymous attempts remain separate from registered user attempts.
* Hidden tags and admin-only survey data remain unavailable to anonymous
  participants.

Add focused tests for:

* admin-only anonymous link creation/disable endpoints
* token lookup and unavailable-token behavior
* anonymous start/answer/complete lifecycle
* hidden-tag isolation
* separation from logged-in user attempts
* reporting representation for anonymous attempts

Manual checks:

* Admin creates an anonymous link for a published survey and copies it.
* Anonymous visitor completes a survey in a fresh/incognito browser.
* Logged-in and anonymous attempts do not merge or overwrite each other.
* Disabled/expired/bad token shows a safe unavailable state.
* Admin Results includes anonymous attempts without exposing participant-only or
  hidden-tag data to the anonymous UI.
