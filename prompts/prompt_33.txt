# Received Feedback

1. Add an admin-only section for standard software version tracking and patch
   notes.
2. Update `markdown/` and other process scaffolding so release notes are
   created as part of the development process.
3. Validate and publish release notes when production deploys happen through
   `npm run deploy` or direct pushes to `main`.
4. Follow-up adjustment: make the release-note workflow more automated for

   promoting that draft into a versioned release note before deployment.

Do not silently drop feedback. Anything not implemented must be moved to

# Intake Classification

* Feature request: admin release-note page and API.
* Deployment/setup issue: release-note validation before production deploy.
* Documentation/process issue: durable workflow updates and phase prompt.
* Security/data-visibility risk: release notes must be admin-only and protected
  server-side.

This phase changes:

* API contracts: adds `GET /api/admin/releases`.
* Authentication or authorization: uses existing admin-only API protection.
* Deployment handling: `npm run deploy` and the main GitHub Actions workflow
  validate release-note freshness.
* Validation/test expectations: adds release parser, endpoint, and script tests
  where practical.

This phase does not change:

* Database schema or migrations.
* Participant-facing survey behavior.
* Hidden-tag visibility.
* Production secret handling.

* `notes/claude_handoff_phase_33.txt` before closeout.

## Non-Goals

Do not implement:

* In-app editing or database storage for release notes.
* Public release notes visible to standard users or anonymous visitors.
* A full changelog generator from git commits.
* Broad admin navigation redesign.
* Release automation that commits files automatically.

## Assumptions

* Root `package.json` is the source of truth for the app release version.
* Release-note Markdown is committed and reviewable.
* Admins view notes in-app; developers create and edit notes in the repo.
* Direct raw `git push origin main` is enforced by GitHub Actions, not local
  hooks on every developer machine.

# Acceptance Criteria

The phase is ready for review when:

* Admins can open `/admin/releases` and see the current root app version plus
  parsed release history.
* Unauthenticated users receive `401` from `GET /api/admin/releases`.
* Authenticated non-admin users receive `403` from `GET /api/admin/releases`.
* `npm run release:notes` can scaffold the current version note without
  overwriting an existing file.
* `npm run release:draft` creates or preserves the unpublished draft note.
* `npm run release:prepare` promotes a completed draft note into a versioned
  release note, bumps the root app version, resets the draft, and refuses
  placeholder draft text.
* `npm run release:check` validates semver, release-note format, and current
  package-version coverage.
* `npm run deploy` runs release validation before pushing.
* GitHub Actions validates release notes before Azure deploy on `main`.
* Durable docs and phase closeout artifacts are updated.
