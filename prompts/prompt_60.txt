# Received Feedback

Implement an issue profile emoji burst:

1. Admins manage emoji metadata on tag catalog definitions.
2. Participants accumulate emoji counts from matched hidden tags.
3. Participants receive only emoji/count aggregates, never hidden tag keys,
   values, categories, labels, or IDs.
4. The Ready to submit screen plays a short emoji burst from the thermometer top.
5. Reduced-motion users get a static collected emoji presentation.

---

# Scope

Implement:

* Add nullable emoji metadata to tag definitions.
* Add participant-safe emoji aggregate to runner responses.
* Compute frequency-based emoji counts from selected option tags, Other tags,
  and matching value tags.
* Add CSS/DOM emoji burst and static collected emoji cluster to the survey
  runner.
* Add focused shared/API/web tests and update phase/release docs.

## Non-Goals

Do not implement:

* Participant-visible tag/category labels.
* Canvas or third-party particle engine.
* Emoji mapping from `game_info/tag_emoji_mapping.txt`.

## Assumptions

* Emoji metadata is managed in the Admin Tags catalog.
* Frequency-based counts are preferred over unique-category counts.
* Burst particle count is capped at 40.
* Emoji-only participant output is acceptable.

## Open Questions

None.

# Acceptance Criteria

The phase is ready for review when:

* Tag definitions can store nullable emoji metadata.
* Runner payloads include only `{ emoji, count }` aggregate items and
  `totalCount`.
* Hidden tag identities remain absent from participant and anonymous payloads.
* Ready to submit triggers one capped emoji burst when emoji counts exist.
* Reduced-motion behavior skips animated particles and keeps static emoji
  collection visible.
