# Received Feedback

The survey-taking interface should no longer show numeric "Question X of Y" or
"Page X of Y" labels. It should show estimated time remaining. The estimate
should respect the participant's visible path, including branching and page
jump behavior.

---

# Intake Classification

* participant-facing UX feature
* shared/client helper logic
* participant API payload consumption
* accessibility and mobile layout risk

Expected impact:

* participant runner display
* participant-safe timing payload usage
* shared remaining-time helper
* tests for estimate calculation and path handling

---

# Scope

Implement:

* Remove participant-facing numeric progress labels from the survey-taking
  interface, including "Page X of Y" and "Question X of Y" style copy.
* Use the Phase 25 effective survey estimate as the total survey estimate.
* Calculate remaining time from the resolved visible page path and remaining
  question weights.
* Respect branching, page-jump, hide-page, and skip behavior by using the
  existing resolved page path helpers.
* Count the current page's unanswered visible questions as remaining.
* Display concise copy such as:
  * "About 3 min remaining"
  * "Less than 1 min remaining"
  * "Almost done"
* Keep the visual progress bar if useful, but avoid numeric page/question copy.
* Keep Admin pagination and Admin workflow labels unchanged.
* Handle authenticated and anonymous participant flows consistently.

## Non-Goals

Do not implement:

* Admin timing override controls
* statistical timing derivation
* per-attempt activity instrumentation
* dynamic estimates based on active elapsed time
* survey navigation changes
* dashboard/admin pagination label changes
* completed/review total elapsed-time display unless explicitly required

## Assumptions

* Phase 25 exposes an effective total estimate to participant-safe payloads or
  a helper can derive it from the participant-safe survey object.
* Remaining-time distribution can use backend/default question-type weights as
  proportions while scaling to the effective survey estimate.
* The estimate is approximate and should use user-friendly rounded copy.

## Open Questions

* Should the completion/review screen show total estimated survey time? If
  unanswered, do not add completion-screen timing beyond any existing summary
  needed for consistency.

---

# Acceptance Criteria

The phase is ready for review when:

* Participant runner shows estimated time remaining instead of numeric progress
  labels.
* Straight survey estimates decrease as the participant advances.
* Branching/page-jump/skip behavior updates the estimate after path changes.
* Anonymous and authenticated runners display timing consistently.
* Admin pagination labels remain unchanged.
* Survey navigation and completion behavior are unchanged.
* Participant UI does not expose Admin timing audit metadata.
