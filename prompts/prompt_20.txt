# Scope

Implement:

* Admin user detail endpoint under the admin API namespace with safe profile
  metadata and survey statistics.
* Admin user detail view linked from the existing admin users table.
* Admin-triggered password reset initiation through the Phase 18 reset service.
* User-facing notification via email service, not admin-visible tokens.
* Clear admin UI feedback for reset initiation.
* Shared admin user detail/reset response types.
* Reuse or deliberately mirror Phase 19 registered-user survey-stat semantics:
  available published surveys, in-progress registered attempts, completed
  registered attempts, last registered activity, and completion rate. Do not
  mix anonymous attempt statistics into registered user detail.
* Keep admin profile demographics read-only; standard users continue to update
  their own profile only through `/api/profile`.

## Non-Goals

Do not implement:

* showing reset tokens, password hashes, or reset links to admins
* account impersonation
* admin editing of demographics unless explicitly needed and approved
* broad CRM/account management workflows
* bulk user actions
* extending `/api/profile` with user-id parameters or admin behavior
* changing the Phase 19 completion-rate definition unless this phase explicitly
  updates both user and admin surfaces consistently

## Assumptions

* Admin user detail remains read-only for profile demographics in this phase.
* Reset initiation sends the same type of reset email as the user-requested
  flow.
* Existing self-role-change protections remain intact.
* Admin detail may show a user's email, first name, last name, role,
  registration timestamp, approved profile metadata, and registered survey
  statistics. Avoid additional PII.
* Admin-triggered reset responses are generic/success-oriented and never include
  reset URLs or token material, even in development/no-op email mode.
* If the target user has no `user_profiles` row, return null profile fields
  rather than creating one during read.
* The Phase 19 profile update API is partial-merge and self-service only;
  admin detail should not rely on request body user ids or mass-assign profile
  fields.

## Open Questions

* Should admins be allowed to reset their own password from admin tools, or only
  through profile/settings? If unanswered, allow the same safe reset initiation
  but never expose the token.
* Should the admin detail label clarify that completion rate is "of all
  available surveys"? If unanswered, keep the Phase 19 calculation and use
  concise UI wording that does not imply "of started surveys."
* Should abandoned attempts appear in "last activity" for admin detail? If
  unanswered, keep Phase 19 behavior: abandoned attempts may set last activity
  but do not contribute to available/in-progress/completed tiles.

# Acceptance Criteria

The phase is ready for review when:

* Admin can open a user detail view from user management.
* Detail view shows safe profile metadata and survey statistics.
* Admin can initiate a password reset for a user without seeing token contents.
* Standard users and unauthenticated requests receive 403/401 as appropriate.
* Password hashes and reset secrets are never returned.
* Admin user detail returns 404 for unknown users without leaking secrets.
* Anonymous attempts do not affect registered user detail statistics.
* The existing `/api/profile` self-service endpoints remain current-user-only.
