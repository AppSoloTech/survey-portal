# Scope

Implement:

* Merriam-Webster Collegiate Dictionary API lookup behind a small provider
  interface so future provider swaps are isolated.
* Environment-driven provider configuration:
  * `DICTIONARY_PROVIDER=disabled` or equivalent safe default.
  * `DICTIONARY_PROVIDER=merriam-webster` for the configured provider.
  * `MERRIAM_WEBSTER_COLLEGIATE_API_KEY` for the provider key.
* Server-side lookup endpoint available only to Admins.
* No client-side direct calls to Merriam-Webster.
* Graceful states for:
  * definition found
  * multiple definitions found
  * spelling suggestions/no exact match
  * provider unavailable/rate-limited
  * provider not configured
* Admin UI that lets the Admin accept, edit, or ignore a suggested definition.
* Store source metadata on glossary entries where the Phase 39 schema supports
  it, such as `manual`, `dictionary_suggested`, provider name, and lookup
  timestamp.
* Tests that mock provider responses instead of calling the live service.

The Admin must remain in control of the saved glossary definition. Suggestions
should never overwrite existing definitions without explicit Admin action.

## Non-Goals

Do not implement:

* Participant inline term rendering.
* Automatic glossary entry creation without Admin confirmation.
* Client-side direct calls to the dictionary provider.
* Live-network tests.
* Storing API keys in source files, migrations, fixtures, snapshots, handoffs,
  or screenshots.
* Rich text or HTML dictionary output.
* Non-English dictionary behavior unless explicitly requested.
* Commercial/provider licensing changes beyond documenting the production
  account/key handoff requirement.

## Assumptions

* The dictionary provider is optional. The glossary must remain usable when the
  provider is not configured.
* Dictionary definitions are suggestions only and may be edited before save.
* Provider responses should be normalized to plain text before reaching the UI.
* Automated tests should use mocked provider payloads and should not require
  the local real API key.

## Open Questions

* Should lookup results be cached? If unanswered, avoid durable caching beyond
  saved source metadata in this phase.
* Should Merriam-Webster branding be displayed in the Admin-only suggestion UI?
  Verify current provider requirements during implementation and comply if
  required.

# Acceptance Criteria

The phase is ready for review when:

* Admins can request a definition suggestion for a glossary term.
* Admins can accept, edit, or ignore the suggested definition.
* No-match and provider-error states are clear and do not block manual entry.
* Non-admin users cannot call the lookup endpoint.
* Dictionary provider secrets are read from environment configuration only.
* Tests do not depend on live network access.
* Participant-facing survey behavior remains unchanged.

Run:

```bash

```

Add focused tests for:

* Admin-only lookup authorization.
* Successful provider response normalization.
* Multiple-definition response normalization.
* No-match/spelling-suggestion fallback.
* Provider error/rate-limit fallback.
* Provider-not-configured behavior.
* Accepting/editing/ignoring a suggested definition in the Admin form.
* No live API calls during automated tests.

Manual checks:

* With provider disabled, Admin can still manually create/edit glossary entries.
* With the configured local provider env vars, Admin can request a suggestion
  and edit it before save.
* No-match terms show manual-entry fallback.
* Existing saved definitions are not overwritten unless the Admin explicitly
  applies a suggestion.
* Mobile pass at 375/768/1280px for the lookup UI.
