# Scope

Implement:

* Make the Results summary area collapsible with an accessible disclosure
  control using `aria-expanded` and clear keyboard behavior.
* Preserve existing summary content, date filters, Refresh, CSV export,
  question stats, option distributions, and hidden-tag rollup semantics.
* Keep Results header actions and date-range controls reachable even when the
  dense summary body is collapsed. Collapsing should hide the noisy report body
  such as counts, question stats, and hidden-tag rollup, not strand core
  controls behind an unclear state.
* Keep the date range filter applied consistently to the summary, CSV export,
  and attempt list.
* Replace the bottom all-attempts rendering with a large-data-friendly attempt
  browser:
  * the API must support bounded server-side pagination for
    `GET /api/surveys/:id/attempts`
  * the API must apply pagination in SQL with `limit`/`offset` and a matching
    filtered count query; do not fetch every attempt and slice in memory
  * the API should default to page 1 and page size 25 when pagination query
    parameters are omitted
  * the shared response should preserve top-level `surveyId` and `attempts`,
    and add pagination metadata, for example:
    `{ page, pageSize, totalCount, totalPages, hasNextPage, hasPreviousPage }`
  * the frontend must request only the visible page
  * the UI must show total count or visible range context
  * the UI must provide previous/next or numbered page controls
  * the UI must provide a conservative page-size choice, such as 25, 50, and
    100 attempts per page
  * the list should use a denser table-like or row layout that remains usable
    at 375/768/1280px widths
* Reset or clamp the current attempts page when date filters or page size change.
  Prefer resetting to page 1 on filter or page-size changes; clamp only when the
  server response proves the requested page is beyond the filtered result set.
* Preserve attempt detail loading by attempt id and existing review-tag editing
  behavior.
* Keep attempt detail as a single detail panel, not content repeated inside
  every paginated row. It may close when the selected attempt falls off the
  current page after filters, pagination, or page-size changes.
* Handle loading, empty, and error states for paginated attempts separately from
  summary loading where practical.
* Avoid refetching the full report summary or tag catalog when only the attempts
  page changes. Refresh and date-range changes may reload both summary and
  attempts.
* Update shared response types and web API helpers for pagination metadata.
* Add focused automated tests for API pagination and Results UI guardrails.
* Update release notes, phase log, and handoff during implementation.

## Non-Goals

Do not implement:

* New charts, analytics definitions, or reporting calculations.
* Materialized report tables, background jobs, queues, or caching layers.
* CSV pagination. CSV export should continue exporting the selected date range.
* Attempt-detail pagination or virtualization inside one selected attempt.
* Search, status filtering, sorting controls, or saved report views unless the
  human explicitly adds them to this phase.
* Changes to review-tag, hidden-tag, `<ALL>`, or category emoji semantics.
* Participant-facing UI changes.
* A third-party data-grid dependency unless the human approves it first.

## Assumptions

* Offset/page-number pagination is acceptable for this admin MVP phase, provided
  the query is bounded and ordered deterministically.
* The existing attempt ordering should remain newest first:
  `started_at desc nulls last, id desc`.
* Page size should default to a modest value, such as 25.
* A selected attempt detail may close when the attempt is no longer visible
  after changing filters/page; preserving it is optional if the UI remains
  understandable.
* The summary disclosure can default to expanded for continuity, but it must be
  easy to collapse and keep the attempts browser prominent.
* This phase completes the "reporting pagination" part of the existing
  `markdown/FOLLOW_UPS.md` item. Hidden-tag filtering remains deferred unless
  explicitly added later.

## Open Questions

None blocking. Status filters, attempt search, and saved report views are useful
future enhancements but are intentionally out of scope for this phase.

# Acceptance Criteria

The phase is ready for review when:

* The Results summary can be expanded and collapsed with an accessible control.
* Collapsing the summary reduces visual noise without removing Refresh, CSV,
  date-range, or reporting capabilities from the page.
* The attempts endpoint validates pagination input and returns a bounded page of
  attempts plus pagination metadata.
* The attempts endpoint does not return every attempt by default.
* The attempts endpoint uses database-level pagination and a filtered total
  count rather than in-memory slicing.
* Attempts remain ordered newest first with deterministic tie-breaking.
* Date-range filters apply to both summary and paginated attempts.
* Changing date range or page size resets/clamps pagination so the page does not
  become empty due only to stale page state.
* The frontend shows visible range/total context and disabled states for
  unavailable pagination actions.
* Changing only attempt page does not refetch report summary or tag catalog.
* Attempt detail fetch, display, and review-tag editing still work from the
  paginated list.
* Empty, loading, and error states remain clear.
* Admin-only authorization remains enforced on report, attempts, detail, review
  tag, and CSV endpoints.
* Participant and anonymous payloads remain unchanged.
* Every received feedback item is fixed, deferred with a reason, or blocked by a
  documented question.

# Testing Requirements

Default full validation gate:

```bash
npm run typecheck
npm run lint
npm run build
npm test
npm run release:check
git diff --check
```

At minimum, add or update focused API tests around:

* attempts pagination default page size and metadata
* custom valid page/pageSize
* omitted pagination query params default to page 1 and page size 25
* invalid page/pageSize validation
* deterministic newest-first ordering
* date-range filtering with pagination metadata
* requested page beyond total pages returns an empty bounded page plus accurate
  metadata, or otherwise follows a clearly tested clamp behavior
* non-admin and unauthenticated access still forbidden
* unknown survey still returns the existing not-found behavior

At minimum, add or update focused web tests/source guardrails around:

* summary disclosure includes `aria-expanded`
* summary collapse leaves core actions/date controls reachable
* attempts fetch includes page/pageSize parameters
* pagination controls and visible range/total copy exist
* date-range or page-size changes reset attempt pagination
* page-only changes do not intentionally refetch report summary or tag catalog
* existing review-tag category `<ALL>` controls remain present

Suggested focused commands before the full gate:

```bash
npm run test -w apps/api -- test/reporting.test.ts
npm run test -w apps/web -- SurveyResultsPage.test.ts
```

Manual checks to perform or document:

* Results page at 375/768/1280px with summary expanded and collapsed.
* Large seeded or manually created attempt set where page 1, next page, previous
  page, and page-size changes behave predictably.
* Date-range changes update both summary numbers and the attempts page.
* CSV export still uses the selected date range.
* Opening attempt detail from a paginated row still shows answers and review-tag
  controls.
* Empty-state behavior for no attempts in a date range.

If a command cannot run, document why and what narrower validation was
completed.
