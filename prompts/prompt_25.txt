completed survey attempts, with a simple Admin override for total estimated
survey completion time.

`prompts/prompt_23.txt`. It should build the durable timing foundation but
should not yet replace participant progress labels.

# Received Feedback

The estimated time to complete a survey should be derived from statistical data
gathered from actual surveys completed, or eventually attempts. Admins should
also be able to override any statistically derived value with their own input.

# Scope

Implement:

* Backend-owned effective survey completion estimate.
* Survey-level statistical estimate from valid completed attempts:
  * use `completed_at - started_at`
  * include only attempts with both timestamps
  * ignore non-positive durations
  * use completed attempts for the same survey only
* Admin override for total estimated survey completion minutes.
* Override precedence:
  * Admin override wins first
  * statistical estimate wins second
  * backend default estimate is last fallback
* Backend default estimate based on current survey question type weights.
* Admin Setup UI showing:
  * derived estimate
  * statistical sample count
  * default estimate
  * effective estimate
  * estimate source: `admin_override`, `statistical`, or `default`
  * override input in minutes
  * clear-override action
* Shared/Admin timing types for:
  * `derivedEstimateSeconds`
  * `defaultEstimateSeconds`
  * `adminOverrideSeconds`
  * `effectiveEstimateSeconds`
  * `sampleCount`
  * `estimateSource`
* Participant-safe survey payload support for only the effective estimate if
  needed by later phases.

Use the next available migration number at implementation time. Do not hard-code
the migration number in the prompt or docs because earlier phases may change the
sequence.

## Non-Goals

Do not implement:

* participant progress label replacement
* running/dynamic time remaining display
* per-page or per-question activity instrumentation
* per-question Admin overrides
* per-question-type Admin override settings
* analytics based on abandoned or in-progress attempts
* live dashboards or historical trend charts
* changes to survey navigation, skip logic, reporting counts, hidden tags, or CSV

## Assumptions

* The first robust estimate is survey-level, not per-question or per-page.
* The first Admin override is total survey minutes.
* Admin override always beats statistical and default estimates.
* Completion duration from `started_at` to `completed_at` is acceptable for this
  foundation even though it may include idle time.
* Timing override is survey metadata, not structural survey content, so it can
  be edited on published surveys.

## Open Questions

* Should very large outlier completed durations be trimmed from the statistical
  estimate? If unanswered, use a conservative cap or median-based approach and
  document it in the implementation handoff.

---

# Acceptance Criteria

The phase is ready for review when:

* Admins can view the effective survey completion estimate on Admin Setup.
* Admins can set and clear a total estimated completion time override.
* Derived estimates use completed attempts for the same survey.
* The effective estimate source is clearly identified to Admins.
* Surveys with no completed-attempt sample use a backend default estimate.
* Non-admin users cannot access Admin timing metadata or override mutations.
* Published surveys can receive timing override metadata without unlocking
  structural edits.
* Participant APIs do not expose Admin audit metadata.
