# Received Feedback

1. Admins need one aggregate report from a suite of tests.
2. The report should explain which service appears to fail first.
3. The report should help decide whether to upgrade App Service, PostgreSQL, or
   investigate endpoint/query behavior.

# Intake Classification

* Admin reporting UI
* API client change
* no database schema change if Phase 52 is complete
* no participant-facing behavior change
* no browser execution controls

# Scope

Implement:

* Add API client helpers for Phase 52 suite endpoints.
* Extend `/admin/performance` or add a sub-view/tab for capacity suites.
* Show recent suites with:
  * suite key
  * status
  * target
  * started time
  * duration
  * profiles included
  * first failing profile/stage/VU
  * bottleneck
  * confidence
* Show suite detail with:
  * recommendation summary
  * first degradation point
  * child run table
  * pass/fail by profile
  * sampled signal highlights
  * Azure metric availability and caveats
  * SQL metric availability and caveats
  * sample limit/window labels when the API returns bounded samples
  * stored suite markdown rendered safely as text/preformatted content
* When samples exist, render simple dense tables or compact sparklines only if
  they fit existing UI patterns. Avoid broad charting dependencies.
* Explain unavailable metrics as unavailable, not zero.
* Treat Phase 54 Azure fields as optional:
  * before Phase 54, show Azure Monitor as unavailable/not collected
  * after Phase 54, show returned availability, time window, granularity, and
    caveats when present
* Link from suite child runs to the existing run detail where practical.
* Add focused tests for formatting, status/confidence labels, unavailable
  metric handling, and source-level guards against browser run controls.
* Add minimal component/render tests for:
  * empty suite state
  * selecting a suite and rendering detail
  * child-run link rendering when a child run id is present
  * unavailable Azure/SQL metric rendering
* Ensure UI rendering does not expose secret-like values if malformed historical
  config/summary/sample payloads contain them.

## Non-Goals

Do not implement:

* Run/stop/schedule/retry/delete buttons.
* Browser k6 execution.
* Azure metric reads from the browser.
* Azure resource scaling controls.
* Download/browse local artifacts.
* Heavy charting libraries unless explicitly approved.

## Assumptions

* Admins want an operational dashboard, not a marketing page.
* The suite report should favor clear caveats over false precision.
* Existing one-off run viewer remains useful and should not be removed.

## Open Questions

* None. Choose the smallest UI that makes suite evidence readable.

# Acceptance Criteria

The phase is ready for review when:

* Admins can inspect recent capacity suites.
* A suite detail shows child runs and the aggregate recommendation.
* First failing profile/stage/VU is visible when present.
* Bottleneck confidence and caveats are visible.
* Missing Azure/SQL metrics are not shown as zero.
* Bounded sample windows/limits are understandable when sample rows are shown.
* No UI control can run, stop, schedule, retry, delete, or scale resources.
* The view is usable at 375/768/1280px.
