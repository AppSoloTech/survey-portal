# Phase 58 — Glossary Search Result To Entry Workflow

## Purpose

Connect Glossary question-search results to the existing create-entry workflow
so Admins can turn discovered question text into reviewed, defined Glossary
terms without automatic entry creation.

Source feedback or planning note:

```txt
notes/client_review_2026-06-30_glossary_question_search.txt
```

This phase completes the first question-search workflow after the
candidate-term behavior is confirmed. The current draft pre-fills the existing
Glossary entry form from the search query tied to a selected result. It should
save through existing Glossary CRUD only.

# Received Feedback

1. Admins should be able to search existing assessment question text and use a
   matching word, phrase, or question text to start a Glossary entry.
2. The selected candidate should become a term to define and later use in the
   existing hover/focus pop-up behavior.
3. The workflow must remain deliberate: search selection should not immediately
   create a Glossary entry.
4. The first implementation should not persist source-question references.
5. The current draft uses the raw search query as the editable prefilled term.
   This must be confirmed against the client's "select text, word, or phrase"
   expectation before implementation starts.

Do not silently drop feedback. Anything not implemented must be moved to
Non-Goals, Deferred Work, or Open Questions.

# Scope

Implement:

* A result action on the Phase 57 question-search result UI.
* On action:
  * use the current search query, trimmed, as the prefilled `canonicalTerm`
  * switch or scroll to the existing create-entry workflow
  * leave definition review and save under Admin control
  * focus the next useful field, preferably the definition field if the term is
    prefilled successfully
  * keep the term editable so Admins can shorten a full-question search into a
    more reusable Glossary term before saving
* Duplicate detection before prefill or save:
  * compare the candidate query against loaded Glossary canonical terms and
    aliases using the existing normalization behavior where available
  * clearly tell the Admin when the candidate already exists
  * avoid creating duplicates client-side, while preserving server-side
    duplicate enforcement as the source of truth
* Temporary source context:
  * show the assessment/page/question context for the selected search result
    near the create form
  * make clear that the context is informational and not saved as a source
    reference
  * clear the context after a successful save, cancel, or reset
* Preserve dictionary lookup behavior from Phase 40 after prefill.
* Save through the existing Glossary create endpoint and existing validation.
* If the create form already contains unsaved non-empty values, ask for
  confirmation before replacing them with the selected search candidate.

## Non-Goals

Do not implement:

* Automatic Glossary entry creation from a search result.
* Persisted source assessment/page/question references.
* Database schema changes.
* New backend create routes.
* Selecting arbitrary substrings from a result card beyond using the current
  query as the candidate term, unless the pre-Phase 58 confirmation changes this
  scope.
* Fuzzy duplicate detection or semantic duplicate detection.
* Participant-facing rendering changes beyond whatever existing Glossary entries
  already cause after being saved.
* Search over answer options, hidden tags, response data, or non-question
  content.

## Assumptions

* Phase 56 and Phase 57 are implemented and available.
* The current query is the desired candidate term, even when the clicked result
  contains a longer question. This assumption must be confirmed before Phase 58
  starts.
* Existing Glossary CRUD remains the authoritative save path.
* Existing server-side uniqueness validation remains the source of truth for
  preventing duplicate match strings.
* Source context should help the Admin remember where the candidate came from,
  but it should not become permanent data yet.

## Open Questions

* Confirm before implementation: should Phase 58 use the current search query as
  the editable prefilled term, or should it support selecting an arbitrary
  word/phrase from inside the result question text?
* Confirm whether the UI should actively caution Admins when the candidate term
  is a full question, since whole-question Glossary entries are usually less
  reusable than shorter terms or phrases.
* Persisted source references and richer duplicate detection should be revisited
  only after the first workflow is in use.

# Implementation Priorities

Handle items in this order unless the human explicitly changes priority:

1. Preserve existing create/edit/archive and dictionary behavior.
2. Prevent accidental overwrites of unsaved create-form data.
3. Prefill the candidate term cleanly and keep Admin review/save deliberate.
4. Provide duplicate warnings without bypassing server validation.
5. Keep source context temporary and clearly non-persistent.

Keep changes narrowly scoped to this phase.

---

# Acceptance Criteria

The phase is ready for review when:

* A search result action can prefill the existing Glossary create form with the
  confirmed candidate text.
* Selecting a result does not immediately create an entry.
* Admins can review or edit the prefilled term, add a definition, use existing
  dictionary assistance, and save through the existing create flow.
* Duplicate candidates are detected against loaded canonical terms and aliases
  and communicated before creation.
* Server-side duplicate validation still handles race conditions or stale loaded
  entry data.
* Temporary source assessment/page/question context is shown after selection and
  is cleared at the appropriate time.
* No persisted source-question reference is added.
* Existing Glossary management behavior is not regressed.
* Every received feedback item is fixed, deferred with a reason, or blocked by a
  documented question.
