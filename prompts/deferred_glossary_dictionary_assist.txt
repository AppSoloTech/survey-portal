# Deferred Glossary Phase — Dictionary-Assisted Glossary Definitions

## Purpose

Add an Admin-only dictionary lookup flow that suggests definitions for glossary
terms before the Admin saves or updates the final glossary definition.

This deferred phase should build on the deferred global glossary Admin
foundation. It should not implement participant inline glossary rendering.

Source feedback or planning note:

```txt

Webster/dictionary API to suggest a definition. If no definition is found, the
Admin should enter their own definition.
```

# Scope

Implement:

* Select and document the approved dictionary/Webster provider after verifying
  current official provider docs, terms, authentication, and response shape.
* Server-side lookup endpoint available only to Admins.
* Environment-driven provider configuration with no secrets committed.
* Graceful states for:
  * definition found
  * multiple definitions found
  * no definition found
  * provider unavailable/rate-limited
  * provider not configured
* Admin UI that lets the Admin accept, edit, or ignore a suggested definition.
* Store source metadata on glossary entries where the global glossary schema
  supports it, such as `manual`, `dictionary_suggested`, provider name, and
  lookup timestamp.
* Tests that mock provider responses instead of calling the live service.

The Admin must remain in control of the saved glossary definition. Suggestions
should never overwrite existing definitions without explicit Admin action.

## Non-Goals

Do not implement:

* participant inline term rendering
* automatic glossary entry creation without Admin confirmation
* client-side direct calls to the dictionary provider
* live-network tests
* storing API keys in source files, migrations, fixtures, or snapshots
* rich text or HTML dictionary output
* non-English dictionary behavior unless explicitly requested

## Assumptions

* The dictionary provider is optional. The glossary must remain usable when the
  provider is not configured.
* Dictionary definitions are suggestions only and may be edited before save.
* Provider responses should be normalized to plain text before reaching the UI.

## Open Questions

* Which dictionary/Webster provider and plan should be used? If not decided,
  implement the client behind a small provider interface and document the
  required environment variables as placeholders.
* Should lookup results be cached? If unanswered, avoid durable caching beyond
  saved source metadata in this phase.

---

# Acceptance Criteria

The phase is ready for review when:

* Admins can request a definition suggestion for a glossary term.
* Admins can accept, edit, or ignore the suggested definition.
* No-match and provider-error states are clear and do not block manual entry.
* Non-admin users cannot call the lookup endpoint.
* Dictionary provider secrets are read from environment configuration only.
* Tests do not depend on live network access.
* Participant-facing survey behavior remains unchanged.

  before save.

  applies a suggestion.
