# Received Feedback

User demographics cleanup: ensure users can update profile with address, phone
number, and full name.

Clarified product decision:

* Full name means users can edit the existing account first name and last name
  from Settings.
* Address should be one optional profile text field.
* Phone number should reuse the existing contact-number behavior where
  practical, with user-facing copy updated to "Phone number".

# Scope

Implement:

* Let authenticated users update their own first name and last name from
  Settings.
* Add a single optional `address` profile field with length validation.
* Present the existing contact number field as "Phone number" in user-facing
  Settings and admin read-only detail.
* Preserve current survey statistics on Settings.
* Preserve the Settings password reset panel and cooldown behavior.
* Keep profile routes self-service only; no `/api/profile/:id` behavior and no
  admin demographic editing.
* Update shared profile/request/response types as needed.
* Update admin user detail to display address and phone read-only if present.

Use the next available migration number at implementation time. Do not hard-code
the migration number in this prompt or docs because earlier phases may change
the sequence.

## Non-Goals

Do not implement:

* Structured address fields.
* Profile photos or file uploads.
* Admin editing of user profile demographics.
* CRM/account-management workflows.
* Additional demographic fields beyond full name, address, and phone number.
* Session invalidation after name/profile edits.
* Password reset behavior changes.

## Assumptions

* First and last name remain on the existing auth user model.
* Address belongs in `user_profiles` as optional self-service profile data.
* `contactNumber` may remain the internal/shared field name if that keeps the
  change smaller; user-facing copy should say "Phone number".
* Empty strings should normalize to null for optional profile fields.

## Open Questions

* None.

# Acceptance Criteria

The phase is ready for review when:

* Users can update first name, last name, address, and phone number from
  Settings.
* Updated names appear after reload and in the account menu/current-user
  identity where applicable.
* Address and phone persist after reload.
* Profile updates remain current-user-only.
* Admin user detail can read address and phone but cannot edit them.
* Existing survey stats and password reset controls still work.
