# Phase 27 — Running Dynamic Time Remaining Instrumentation

## Purpose

Add lightweight participant attempt activity timing so future time remaining
estimates can improve beyond coarse `started_at` to `completed_at` survey
duration.

This phase should build on the statistical foundation from Phase 25 and the
participant display from Phase 26. It should collect safer active-time data but
should avoid overfitting a complex prediction model in the same pass.

Source feedback or planning note:

```txt

the running and dynamic estimated time to survey completion.
```

# Received Feedback

Survey timing should have two parts: the estimated time it would take to
complete a survey, and a running dynamic estimated time to completion while the
participant is taking the survey. The dynamic estimate needs better attempt
activity data than coarse attempt start and completion timestamps.

# Scope

Implement:

* Lightweight activity timing for participant attempts.
* Track authenticated and anonymous participant activity without exposing Admin
  metadata to participants.
* Record enough context to support future dynamic remaining-time estimates, such
  as attempt id, page id, visible question ids or current question id, event
  timestamp, and event type.
* Use capped active-time gaps so overnight pauses or long idle browser sessions
  do not count as continuous active survey time.
* Provide backend helpers that can aggregate active seconds by attempt and by
  survey.
* Keep instrumentation safe for anonymous attempts by validating the
  anonymous-link id and attempt access token.
* Avoid noisy writes where possible, such as by throttling heartbeats or only
  sending events on page entry, answer save, resume, and completion.
* Document how this data should feed a later dynamic estimate model.

Use the next available migration number at implementation time. Do not hard-code
the migration number in the prompt or docs because earlier phases may change the
sequence.

## Non-Goals

Do not implement:

* a complex prediction model
* Admin dashboards for activity timelines
* per-question Admin timing overrides
* exporting timing telemetry to CSV
* changing survey completion validation or navigation
* storing raw text answers in timing events
* participant exposure of timing event internals

## Assumptions

* Running estimates require activity instrumentation before they can become
  meaningfully dynamic.
* Idle gaps should be capped before contributing to active-time statistics.
* Activity timing is operational metadata and should not affect response
  storage, reporting counts, hidden tags, or CSV.
* Completed and abandoned/in-progress attempts can eventually inform dynamic
  models, but this phase focuses on data quality and safe collection.

## Open Questions

* What idle-gap cap should be used? If unanswered, choose a conservative cap,
  document it, and centralize it in the backend timing helper.

---

# Acceptance Criteria

The phase is ready for review when:

* Authenticated participant activity events are recorded for owned attempts.
* Anonymous participant activity events require valid anonymous attempt access.
* Long idle gaps are capped in active-time aggregation.
* Activity helpers can summarize active seconds for an attempt and survey.
* Activity writes do not expose Admin-only or timing-internal metadata to
  participant payloads.
* Survey answering, completion, reporting, CSV, hidden tags, and navigation
  behavior remain unchanged.
