# Received Feedback

The audit identified areas that require browser and assistive-technology
verification for the public and registered-user application:

1. Contrast across light/dark themes and component states.
2. Keyboard-only flows across public, anonymous participant, and registered
   participant routes.
3. Screen reader checks for route changes, survey questions, modals, form
   errors, dashboard cards, and public directory cards.
4. Mobile screen reader/touch checks.
5. Repeatable axe/Playwright-style checks if practical.
6. Documentation for accessibility primitives and future regression checks.

# Scope

Implement:

* Create a durable accessibility test/checklist document, for example
  `markdown/ACCESSIBILITY_TEST_PLAN.md`, covering:
  * keyboard-only workflows
  * screen reader workflows
  * mobile screen reader/touch checks
  * contrast checks in light and dark theme
  * modal, route, auth, account, dashboard, public directory, and survey runner
    forms
  * accepted limitations of automated testing
* Add a concise accessibility primitive guide if new primitives from Phases
  42-45 need usage rules.
* Add automated accessibility checks if practical with the existing stack.
  Prefer a small Playwright plus axe-style smoke suite only if it can run
  reliably in this repo and the human developer approves any new dependency.
* If adding automated browser checks, cover representative routes and states:
  * public home/login/register
  * dashboard
  * category survey list
  * account settings
  * survey attempt with multiple question types
  * anonymous survey route
  * anonymous survey directory
  * light and dark themes
* Add or document contrast verification for:
  * normal text
  * muted text
  * buttons
  * focus rings
  * disabled controls
  * status pills
  * toasts/alerts
  * survey cards, progress indicators, glossary triggers, and modal surfaces
* Update `markdown/REVIEW_CHECKLIST.md` with compact accessibility review
  gates if the team wants them to remain durable.
* Keep any discovered defects either fixed if tiny and in scope, or added to
  `markdown/FOLLOW_UPS.md` with clear phase placement.

## Non-Goals

Do not implement:

* Large UI remediation already owned by Phases 42-45.
* Admin-only accessibility remediation, including Admin builder, Admin results,
  Admin users, Admin custom selects, and Admin drag-and-drop.
* New user-visible accessibility features beyond tiny fixes found while
  validating.
* A heavy end-to-end testing platform unless explicitly approved.
* Live external service dependency in tests.
* Real client data fixtures.

## Assumptions

* Automated accessibility tools cannot replace manual assistive-technology
  testing.
* Some checks may remain manual if browser automation is not available in the
  project environment.
* Documentation is an acceptable deliverable for checks that cannot be
  automated safely.

## Open Questions

* Is the human developer willing to add Playwright and an axe integration as
  dev dependencies? If not, create the manual test plan and add lightweight
  non-browser tests only.

# Acceptance Criteria

The phase is ready for review when:

* The project has a clear repeatable accessibility test plan.
* Existing review workflow includes public/user accessibility checks for future
  phases.
* Contrast testing expectations are documented for both themes.
* Any automated checks added are reliable, documented, and included in an
  appropriate npm script.
* Manual browser/AT gaps are explicitly recorded.
* Deferred defects discovered during validation are added to
  `markdown/FOLLOW_UPS.md`.
