# Phase 23 — Superseded: Estimated Time Remaining Foundation

## Status

Do not implement this prompt as originally written.

The original Phase 23 idea was a frontend-only participant runner change that
would replace numeric progress labels with a simple estimated time remaining
value based on code-configured question-type averages. Client review changed
the product direction to backend-owned statistical timing with Admin overrides
and a later running/dynamic estimate.

The robust timing work is larger than a single participant display pass, so it
has been split into staged follow-up prompts:

* `prompts/prompt_25.txt` — Survey Completion Time Stats And Admin Override
  Foundation
* `prompts/prompt_26.txt` — Participant Estimated Completion Display
* `prompts/prompt_27.txt` — Running Dynamic Time Remaining Instrumentation

Keep this file as a historical marker so future phase planning does not
accidentally revive the smaller frontend-only estimate scope.

---

# Superseded Scope

The old scope would have:

* removed participant-facing "Page X of Y" and "Question X of Y" labels
* added a reusable estimate helper based on remaining visible
  path/questions
* used configurable-in-code averages by question type with a fallback average
* avoided database or backend API changes
* avoided analytics based on actual completion time

That is no longer the intended direction for this feature.

---

# Current Product Direction

Estimated survey completion time should be backend-owned and derived from real
survey attempt data where possible. Admins should be able to override a
statistically derived value with their own total estimated completion time.

The product direction has two separate timing concepts:

* estimated time to complete a survey before or at the start of an attempt
* running estimated time remaining while a participant is taking the survey

The first robust pass should use survey-level completed-at minus started-at
statistics from completed attempts, with a backend default estimate when no
statistical sample exists. Admin override values should win over statistical
and default values.

---

# Notes For Future Implementation

* Do not use a frontend-only hard-coded estimate as the long-term source of
  truth.
* Do not expose Admin statistical audit fields to participant APIs unless a
  prompt explicitly calls for them.
* Treat Admin total-time override as survey metadata, not structural survey
  content, so it can be edited on published surveys.
* Do not squeeze running/dynamic activity instrumentation into the first
  statistical foundation phase.
