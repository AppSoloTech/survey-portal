# Phase 41 - Participant Inline Glossary Rendering

## Purpose

Render Admin-approved global glossary definitions inline in participant-facing
survey question text.

This phase should build on Phase 39, the global glossary Admin foundation.
Phase 40 dictionary assist is helpful but not required if definitions already
exist manually. This phase should make glossary terms discoverable to survey
takers without changing survey logic or response data.

# Received Feedback

Glossary definitions should apply globally to all questions where strings in
the question match predefined glossary terms. The popup the survey user sees
should be dynamic and generated inline just before text is displayed to the
user.

# Scope

Implement:

* Participant-safe glossary payload containing enabled glossary match strings
  and definitions, with no Admin-only metadata.
* A deterministic text-matching helper for plain text question prompts.
* Inline rendering that wraps matched glossary terms with an accessible
  definition trigger/popover.
* Hover, focus, keyboard, click/tap, and mobile-friendly behavior.
* Longest-match-first handling so longer aliases win over shorter overlapping
  matches.
* Case-insensitive matching while preserving the original question text casing.
* Overlap prevention so generated glossary spans are never nested.
* Word/string boundary rules that avoid surprising partial-word matches unless
  a match string explicitly requires punctuation or spacing.
* Admin preview behavior consistent with the participant runner where
  practical.
* Focused tests for matching and rendering behavior.

Apply glossary rendering only to plain participant-facing question text in this
phase unless implementation inspection shows another already-shared prompt
renderer should be covered at the same time.

## Non-Goals

Do not implement:

* Changes to answer validation, skip logic, hidden tags, reporting, CSV exports,
  or response storage.
* HTML/rich-text parsing of question text.
* Glossary rendering inside answer option labels unless explicitly requested.
* Matching inside Admin-only forms while editing.
* Automatic dictionary lookups.
* Analytics for glossary popover opens.
* Participant exposure of Admin-only glossary metadata.

## Assumptions

* Matching strings are plain text, not regular expressions.
* Glossary definitions are plain text.
* Participant rendering can be computed client-side from a participant-safe
  glossary payload.
* If two glossary entries have the same match string, validation from Phase 39
  should prevent ambiguity before this phase.

## Open Questions

* Should glossary matching apply to survey/page descriptions as well as
  question prompts? If unanswered, start with question prompts and leave
  descriptions as a documented follow-up.
* Should answer option labels be matched? If unanswered, do not match option
  labels in this phase.

# Acceptance Criteria

The phase is ready for review when:

* Participant question text automatically marks enabled glossary terms.
* Hover/focus/click/tap opens a definition popover without changing survey
  state.
* Keyboard users can discover, open, and dismiss definitions.
* Mobile users can open and dismiss definitions without layout breakage.
* Matching is deterministic for aliases, casing, repeated terms, and overlaps.
* Disabled glossary entries do not render.
* Admin-only glossary metadata is not exposed to participant APIs.
* Survey validation, navigation, reporting, CSV, hidden tags, and response data
  remain unchanged.

# Testing Requirements

Run:

```bash
npm run typecheck
npm run lint
npm run build
npm test
git diff --check
```

Add focused tests for:

* No matches.
* Single match.
* Repeated matches.
* Case-insensitive matching with original text preserved.
* Aliases and canonical terms.
* Longest-match-first overlap resolution.
* Boundary behavior for partial words.
* Disabled glossary entries excluded.
* Participant payload excluding Admin-only fields.
* Popover accessibility states if component tests exist nearby.

Manual checks:

* Admin creates glossary entries, publishes/uses a survey question containing
  matching text, and participant sees inline glossary affordances.
* Hover/focus/click/tap open and close definitions.
* Keyboard-only pass: Tab to term, open/read/dismiss, continue survey.
* Repeated terms and overlapping terms render cleanly.
* Disabled glossary entry stops rendering without editing the question text.
* Survey answers, completion, Results, and CSV remain unchanged.
* Mobile pass at 375/768/1280px.
