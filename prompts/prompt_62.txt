# Received Feedback

1. When adding a tag to a category that already has an emoji associated with it,
   the new tag does not inherit that emoji.
2. New tags matching an existing category should inherit the category emoji.

Do not silently drop feedback. Anything not implemented must be moved to
Non-Goals, Deferred Work, or Open Questions.

Classification:

* bug or regression
* admin workflow polish

Expected change areas:

* Database schema or migrations: none expected.
* API contracts: no new endpoint expected; create-tag behavior may change.
* Participant-facing survey behavior: indirect only. Participants may receive

* Hidden-tag visibility: preserve existing participant/anonymous privacy

* Authentication or authorization: no model change; tag catalog mutation routes
  remain Admin-only.
* Deployment or secret handling: none.
* Validation/test expectations: yes, add focused regression coverage for new
  tag creation after a category emoji exists.

Terminology note:

* In the Admin Tags UI, "Apply emoji to tag category" currently groups by
  `tag_definitions.tag_key`, not by the visual `tag_groups` section.
* For this phase, "matching an existing category" means a new tag with the same
  normalized `tagKey` as existing catalog definitions that share a category
  emoji.

Ask for clarification only if implementation proves that the product owner meant
visual tag groups instead of tag-key categories.

# Scope

Implement:

* When an Admin creates a new tag definition with a `tagKey` that already has a
  single common non-null emoji across existing catalog entries, the new tag
  inherits that emoji if the create request does not provide a non-blank emoji.
* Preserve explicit per-tag emoji entry on create. If the Admin provides a
  non-blank emoji, use that value instead of inherited category emoji.
* Do not infer an emoji when the matching category has no emoji or has multiple
  conflicting non-null emoji values.
* Preserve the existing category-emoji action that applies or clears an emoji
  across all current definitions for a `tagKey`.
* Ensure the created tag response and subsequent catalog fetches include the
  inherited emoji.
* Add focused automated regression tests for inherited, explicit override,
  cleared/no-emoji, and mixed-emoji cases where practical.
* Update release notes, phase log, and handoff during implementation.

## Non-Goals

Do not implement:

* A new emoji column on `tag_groups`.
* A redesign of tag groups, category naming, or the Admin Tags page.
* Any changes to Phase 61 virtual `<ALL>` bindings except incidental regression
  safety if the touched tag creation path requires it.
* Participant-visible hidden tag/category metadata.
* Changes to emoji burst animation, thermometer scoring, or issue-profile copy.
* Backfilling historical tags beyond the existing category-emoji action.

## Assumptions

* `tagKey` is the category identity used by the issue-profile emoji bulk action.
* Blank, null, or omitted emoji on create means "use the category default if one
  can be inferred."
* A non-blank emoji on create is an intentional per-tag override.
* If a category emoji was cleared, future matching tags should not inherit an
  old emoji.
* If existing matching tags have mixed emoji values, the server should avoid
  guessing and leave the new tag's emoji blank unless the request explicitly
  provided one.

## Open Questions

None blocking. If manual QA shows the client meant visual `tag_groups` rather
than `tagKey` categories, stop and revise this prompt before implementing a
schema change.

# Acceptance Criteria

The phase is ready for review when:

* After applying an emoji to a tag-key category, creating a new tag with the
  same `tagKey` and a blank/null/omitted emoji stores the inherited emoji.
* Creating a new tag with the same `tagKey` and an explicit non-blank emoji
  stores the explicit emoji.
* Creating a new tag for a category with no common emoji leaves emoji null.
* Creating a new tag for a category with mixed existing emoji values does not
  guess an inherited emoji.
* Clearing a category emoji means future matching tags do not inherit the old
  emoji.
* Existing Admin-only authorization and validation behavior for `/api/tags`
  remains intact.
* Participant and anonymous survey payloads still expose only participant-safe
  emoji/count aggregates, never hidden tag identities.
* Phase 61 category `<ALL>` behavior still has focused coverage or is not
  affected.
* Every received feedback item is fixed, deferred with a reason, or blocked by a
  documented question.

  explicit value is kept.

  old emoji.
* Confirm the participant runner still receives only emoji/count aggregates and
  no hidden tag metadata.

If a command cannot run, document why and what narrower validation was
completed.
