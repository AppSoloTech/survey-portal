# Received Feedback

Users should be able to request a password reset from a logged-in state via
profile/settings and from the login screen with "Forgot password?".

---

# Scope

Implement:

* Password reset token table or equivalent relational model storing hashed
  tokens, expiry, consumed timestamp, and user association.
* Request endpoint from login that always returns a generic success response.
* Complete endpoint that verifies a token, hashes the new password with bcrypt,
  consumes the token, and rejects reuse.
* Logged-in profile/settings reset initiation using the same reset service.
* Phase 16 email service integration for reset email dispatch.
* Development-safe reset testing path that does not log reset secrets in
  production.
* Rate limiting consistent with existing auth protection.

## Non-Goals

Do not implement:

* unrelated profile redesign beyond the reset entry point
* admin-triggered password reset; that belongs to Phase 20
* plaintext token storage
* production reset-token logging
* account recovery questions or MFA

## Assumptions

* Password reset responses must not reveal whether an email exists.
* Tokens are single-use and expire.
* Resetting a password should not expose password hashes or reset secrets in any
  API response.

## Open Questions

* Should existing login sessions be invalidated after password reset? If no
  token revocation model exists, document the tradeoff and keep scope minimal.

---

# Acceptance Criteria

The phase is ready for review when:

* Login screen has a "Forgot password?" flow.
* Logged-in user can initiate a reset from profile/settings.
* Existing and non-existing emails receive the same public request response.
* A valid reset token works once and then cannot be reused.
* Expired, consumed, malformed, or unknown tokens fail safely.
* New passwords use existing bcrypt hashing and validation rules.

# Testing Requirements

Run:

```bash

git diff --check
```

Add focused tests for:

* request flow enumeration resistance
* token hash storage and single-use consumption
* expired/used/invalid token failure
* successful password update and old password rejection
* logged-in reset initiation auth requirement
* rate limiting where practical

Manual checks:

* Request reset for existing and non-existing email; both show generic success.
* Use reset link once; confirm new password works and old password fails.
* Reuse expired/consumed token; confirm safe error.
* Trigger reset while logged in from profile/settings.
* Confirm no plaintext reset token is stored in the database.
