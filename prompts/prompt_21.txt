# Received Feedback

For choice-based questions, admins can enable an "Allow Other" setting. When
enabled, the survey-taking UI should render an "Other" option with a free-text
input field. The "Other" option is system-generated, not a normal manually
entered answer option, and the response should be stored separately from
standard option IDs.

# Scope

Implement:

* Question-level `allowOther` setting for `single_select` and `multi_select`.
* Separate response storage for Other selection and custom text.
* Shared API/request/response types for Other answer data.
* Admin question editor toggle for applicable types only.
* Participant "Other" choice plus text input when enabled.
* Required validation:
  * single-select required is satisfied by exactly one standard option or a
    non-blank Other response.
  * multi-select required is satisfied by at least one standard option or a
    non-blank Other response.
  * Other text is required when Other is selected.
* Reporting and CSV display of Other text separately from standard options.

## Non-Goals

Do not implement:

* a normal `answer_options` row for Other
* hidden tags on Other
* conditional logic triggered by Other
* Other support for `scale` unless explicitly approved
* new question types
* any changes to Phase 20 admin user tools, admin password reset, user profile
  contact fields, or `/api/profile`
* cleanup of the old Phase 19 profile columns; that is tracked separately in
  `markdown/FOLLOW_UPS.md`

## Assumptions

* Other applies to `single_select` and `multi_select` only.
* Other text should be trimmed and bounded like other user-entered text.
* Existing responses remain valid after migration with Other disabled by
  default.
* Phase 21 migrations should be additive and preserve existing Phase 20
  migration order.
* Other answer text is participant-entered response data, not admin profile or
  contact metadata.

## Open Questions

* Should reports aggregate identical Other text values? If unanswered, display
  raw Other text per response/export and defer aggregation.

---

# Acceptance Criteria

The phase is ready for review when:

* Admins can enable/disable Allow Other on applicable draft questions.
* Participant UI renders Other only when enabled.
* Standard option, Other-only, and mixed multi-select submissions save
  correctly.
* Other text is not represented as an answer option ID.
* Reporting and CSV show Other text separately.
* Hidden tags remain attached only to real answer options.
