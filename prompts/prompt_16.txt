---

# Scope

Implement:

* Provider-agnostic internal email service interface.
* A disabled/no-op adapter that is safe for local development.
* Placeholder-only environment configuration and `.env.example` entries.
* Template shape/types for future password reset and anonymous survey invite
  emails.
* Tests proving the email client can be initialized disabled and cannot send
  real mail without explicit provider configuration.

## Non-Goals

Do not implement:

* password reset flows
* anonymous survey links or invitation sending
* real email provider integration without credentials/provider approval
* new user-facing routes
* production secret values
* queues, workers, or microservices

## Assumptions

* Provider is unknown, so the service must be small and provider-agnostic.
* `EMAIL_ENABLED=false` or equivalent should be the safest local default.
* Production must not log tokens, passwords, reset links, or full message bodies.

## Open Questions

* Which provider should be used later, such as Azure Communication Services,
  SendGrid, SMTP, or another approved service?

---

# Acceptance Criteria

The phase is ready for review when:

* The API can start with email disabled.
* Placeholder environment variables are documented without secrets.
* The no-op/dev adapter can be exercised safely by tests.
* Future phases can request a typed password reset or anonymous survey invite
  email without choosing a provider in this phase.
* No password reset or anonymous survey functionality is available to users yet.
