# Phase 61 — Category All Review Tags

## Purpose

Add an admin-only convenience tag for every tag category so an admin can apply
all review tags from a category to a text answer in one dropdown action.

# Received Feedback

1. Add a default tag for each current and future tag category.
2. The default tag's name/key should be the category name.
3. The default tag's value should be exactly `<ALL>`.
4. The default category tag must not show on the tag management side of the app.
5. The default category tag should exist in the answer review tag dropdown.
6. When an admin applies the `<ALL>` tag to an answer, all tags within that
   category should be applied to that answer.

# Design Decision

Use a virtual `<ALL>` selector, not persisted `tag_definitions` rows.

The selector should be synthesized from current `tag_groups` and normal
category tag definitions in the answer review-tag dropdown. Selecting it should
call an admin-only bulk action that stores a category-level binding for the
answer and inserts the category's current real `tag_definition_id` values into
`response_answer_tags`.

Do not create, backfill, sync, delete, or hide real `<ALL>` rows in
`tag_definitions`. The virtual approach avoids schema churn, category-delete
orphaning, seed drift, reporting/CSV leakage, and accidental exposure in other
tag catalog queries.

Persist the binding separately so future tag definitions added to that category
propagate to answers where the category `<ALL>` selector was already applied,
including completed/submitted attempts.

# Scope

Implement:

* Add one virtual category `<ALL>` selector for each current and future tag
  category in the answer review-tag dropdown.
* Add a visible virtual `<ALL>` value to builder hidden-tag add and existing tag
  forms after a tag category is selected, including answer option tags, Other
  tags, and text/integer value tags.
* Render the virtual selector under its category using clear admin-facing copy
  that includes the visible value: `<ALL> - Apply all in {category}`.
* Represent the virtual selector internally without a persisted tag definition,
  for example with a sentinel dropdown value such as `category-all:{groupId}`
  or an explicit category bulk-apply API call.
* When the admin selects a category `<ALL>` option for an answer, apply every
  normal tag definition currently in that category to the answer.
* When the admin selects hidden-tag value `<ALL>` in the builder, create a
  living auto-apply subscription for that option/question target and expand the
  current normal tag values in that category.
* When Admin Tags values are created, edited, or deleted later, propagate that
  change to hidden-tag option/question targets with a matching `<ALL>`
  subscription while preserving manually added hidden tags.
* Treat `<ALL>` as a living category binding:
  * applying it stores that this answer should receive all tags in the category
  * if tags are added or moved into the category later, previously bound answers
    automatically gain those new real review tags
  * if tags are moved out of the category later, inherited-only copies are
    removed from bound answers
  * manually added tags are preserved even if the same tag was also inherited
  * submitted/completed attempts are included in this propagation.
* Make the expansion idempotent:
  * already-applied tags remain applied
  * missing category tags are inserted
  * duplicate response-answer tags are not created
  * if the category `<ALL>` binding already exists for that answer, the `<ALL>`
    option is hidden for that category.
* Do not persist `<ALL>` itself as an answer review chip. The intended result is
  that the answer shows the expanded real tags.
* Show a clear `Auto-applying all in {category}` binding indicator with a
  `Stop` action. Stopping auto-apply removes inherited category tags that are
  not also manual tags and prevents future propagation for that category.
* Keep ungrouped tags unchanged; there is no `<ALL>` default for the virtual
  Ungrouped section unless the product owner requests one later.
* Update release notes, phase log, and handoff.

## Non-Goals

Do not implement:

* Participant-visible tags, category names, or tag metadata.
* Changes to issue-profile progress or emoji burst logic.
* Bulk remove for a category.
* A category `<ALL>` tag for the virtual Ungrouped catalog section.
* Real `<ALL>` rows in `tag_definitions`, including an `is_system` discriminator
  or migration/backfill for system tags.
* CSV/reporting semantics beyond showing the expanded real review tags already
  supported by the existing review-tag model.

## Assumptions

* "Tag name" maps to the current `tag_key` field.
* "Category" maps to `tag_groups.name`.
* `<ALL>` is a virtual convenience selector, not a normal managed catalog tag.
* Selecting `<ALL>` should apply only normal tag definitions in the category,
  not a marker tag.
* A category with no normal tags may still be bound so future tags in that
  category propagate to the answer.
* Category creation and rename need no `<ALL>` tag-row handling because the
  selector is synthesized from the current category list.
* Category deletion removes the binding, inherited source rows, and
  inherited-only expanded review tags; manual review tags remain.
* Moving a tag out of a bound category removes inherited copies from bound
  answers but preserves manually added copies.

## Open Questions

None. Product decision: future tags added to a category must propagate to all
answers where that category `<ALL>` selector was applied, including submitted
answers.

# Acceptance Criteria

The phase is ready for review when:

* No real `<ALL>` tag definitions are created in the database.
* Category `<ALL>` bindings are persisted separately from `tag_definitions`.
* The Admin Tags management page and `/api/tags` catalog behavior remain
  unchanged except for any strictly necessary data needed by the review-tag
  dropdown.
* Creating, renaming, or deleting a category automatically affects the virtual
  dropdown choices because they are derived from current categories.
* The answer review-tag dropdown shows `<ALL> - Apply all in {category}` for
  categories that are not yet bound to that answer, including categories whose
  current tags were already applied individually.
* The answer review-tag dropdown hides the category `<ALL>` selector only after
  the category binding exists for that answer.
* Selecting a category `<ALL>` option applies all unapplied normal tags in that
  category to the answer and returns the refreshed review tags.
* New tags created, edited, or moved into a bound category are automatically
  applied to all bound answers, including completed/submitted attempts.
* Tags moved out of a bound category are automatically removed from bound
  answers unless they were also manually applied.
* Admins can stop category auto-apply for an answer; inherited-only tags from
  that category are removed, manual tags remain, and future category tags no
  longer propagate.
* Applying `<ALL>` is idempotent and never creates duplicate
  `response_answer_tags` rows.
* `<ALL>` never appears as an answer review chip, reporting aggregate, CSV
  value, tag catalog row, or Admin Tags management item.
* Existing authorization rules still require an admin and a valid answered text
  response.
* Participant and anonymous payloads still do not expose admin review tags,
  tag catalog entries, or hidden tag metadata.
* Every received feedback item is fixed, deferred with a reason, or blocked by
  a documented question.

  because none exist.

  appears under categories that are not yet bound to the answer.
* Questions tab hidden-tag add and existing tag forms: `<ALL>` appears in the
  value dropdown after selecting a catalog category and expands into the
  category's real values.
* Adding a new Admin Tags value after `<ALL>` expansion updates existing
  marker-backed hidden-tag targets and submitted-attempt admin reporting.
* Selecting `<ALL>` on a text answer expands into the category's real tags.

  duplicates and leaves existing tags intact.

  produce the expected virtual dropdown and future-propagation behavior.

  action; Stop auto-apply is the removal path for auto-applied category tags.
