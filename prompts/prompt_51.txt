# Phase 51 — Admin Performance Report Viewer

## Purpose

Add a read-only Admin page for reviewing performance-test runs persisted by the
CLI harness.

Admins should be able to inspect recent runs, compare key metrics, and read the
stored recommendation. They should not be able to start, stop, or schedule
tests from the browser in this phase.

# Received Feedback

1. Performance-test results should be visible to admins.
2. The feature should avoid extra Azure work.
3. The Admin UI should be a report surface only.
4. Results should be metric-driven and useful for understanding capacity and
   upgrade signals.

# Intake Classification

* Admin reporting feature
* frontend UI change
* API client change
* no database schema change if Phase 49 is complete
* no participant-facing behavior change

# Scope

Implement:

* Add an Admin API client for the Phase 49 endpoints.
* Add an Admin page at `/admin/performance`.
* Link the page from the existing Admin tools panel.
* Show a recent runs table with:
  * scenario
  * status
  * target
  * started time
  * duration
  * max VUs
  * peak requests/sec
  * p95/p99 latency
  * error rate
  * bottleneck
* Show a detail view or expandable panel for one selected run with:
  * high-level metric cards
  * recommendation
  * bottleneck explanation
  * config/summary highlights
  * status coverage for `running`, `completed`, `failed`, and `aborted`
  * CLI configuration highlights such as profile, VUs, duration, ramping
    stages, app DB pool max, and app instance count when present
  * SQL/Azure metric availability labels that distinguish unavailable metrics
    from zero values
  * stored markdown report rendered safely as plain/preformatted text or a
    conservative internal formatting component
* Show concise operator caveats in the detail view when applicable:
  * DB pool and app instance settings affect bottleneck classification.
  * Active connection stats can undercount unless the load-test role has
    `pg_monitor` / `pg_read_all_stats` or uses the app DB role.
  * Recommendations are capacity signals, not automatic upgrade commands.
* Handle loading, empty, error, and not-found states.
* Keep the page operational and dense, consistent with current Admin screens.
* Add route wiring and update route-title behavior if the current app pattern
  expects it for new admin pages.
* Add focused frontend tests for API helpers or deterministic rendering logic
  where practical.

## Non-Goals

Do not implement:

* A "Run test" button.
* Stop, schedule, retry, or delete controls.
* Browser-driven direct DB access.
* Browser-driven k6 execution.
* Azure runner orchestration.
* New Azure resources.
* Raw k6 event browsing.
* Browsing, downloading, or linking local `loadtest/reports/` artifacts from
  the Admin UI.
* Time-series charts unless Phase 49 added a samples table and the human
  explicitly approves chart scope.
* Public or participant-facing UI.

## Assumptions

* Phase 49 provides read-only Admin endpoints.
* Phase 50 persists enough summary data to make the first UI useful.
* Missing Azure Monitor metrics should be displayed as unavailable, not zero.
* Missing PostgreSQL/SQL metrics should be displayed as unavailable or
  permission-limited, not zero.
* The markdown report is internal operational text and should not use unsafe
  HTML rendering.
* The page should explain the "CLI generated / Admin viewed" boundary through
  concise operational copy, without turning into a landing page.
* The UI should display only database-persisted fields. Local CLI artifacts
  under `loadtest/reports/` and ignored manifests remain outside the app.

## Open Questions

* Should admins need export/download links for local artifacts? Deferred for
  this phase because artifacts live outside the app in this trimmed version.

# Acceptance Criteria

The phase is ready for review when:

* Admins can navigate to `/admin/performance`.
* Standard users cannot access the page.
* The page lists recent performance runs from the API.
* The page shows useful metric summaries and recommendations.
* A selected run shows detail without exposing unsafe HTML.
* Empty and error states are clear.
* Missing optional resource metrics are labeled unavailable.
* All four persisted statuses (`running`, `completed`, `failed`, `aborted`)
  render with clear text labels.
* Detail content surfaces persisted configuration and summary caveats without
  implying the browser can run or repair tests.
* No UI control can start, stop, schedule, or generate a load test.
* The page is usable at 375/768/1280px.
