# Received Feedback

Update tag nomenclature. "Tag key" should be "Tag category". "Tag value"
nomenclature can remain as the other side of the key pairs.

---

# Intake Classification

* UX copy polish
* admin workflow terminology alignment
* documentation cleanup

Expected impact:

* Admin tag catalog labels and helper copy
* Survey builder hidden-tag form labels and duplicate messages
* Results/CSV/admin-visible hidden-tag display copy where applicable
* Documentation and prompt references to user-facing tag terminology
* Focused tests that assert visible validation/copy

No database migration, API contract change, participant-facing survey behavior,
hidden-tag visibility change, reporting-count change, or CSV data-shape change
should occur in this phase.

---

# Scope

Implement:

* Replace visible "Tag key" copy with "Tag category" in admin-facing UI.
* Keep "Tag value" copy as-is.
* Update helper text that describes tag key/value pairs to say category/value
  pairs where visible to admins or users.
* Update validation/error text exposed through the API only when the message is
  user-facing; keep request fields named `tagKey` and `tagValue`.
* Update docs and tests that describe user-facing tag terminology.
* Preserve all existing tag data, catalog behavior, duplicate detection, hidden
  tag rollups, and CSV values.

## Non-Goals

Do not implement:

* Renaming `tagKey` in shared/API types.
* Renaming database columns such as `tag_key`.
* Changing request or response payload shapes.
* Changing hidden-tag behavior, visibility, reporting counts, CSV columns, or
  participant payloads.
* Tag grouping; that belongs to Phase 29.

## Assumptions

* "Tag category" is a terminology replacement for the user-visible concept
  currently called "Tag key".
* Internal naming remains stable to avoid needless migration and compatibility
  risk.
* Tests may continue using `tagKey` as a field name, but visible strings should
  assert "Tag category" where practical.

## Open Questions

* None.

# Acceptance Criteria

The phase is ready for review when:

* Admin tag catalog creation/editing labels say "Tag category" and "Tag value".
* Survey builder hidden-tag, value-tag, and Other-tag forms use "Tag category".
* Admin Results display and helper copy no longer refer to "Tag key" where the
  term is visible to users/admins.
* API/database/shared fields remain `tagKey`, `tagValue`, and `tag_key`.
* Existing tag behavior and reports are unchanged.
* Docs and follow-ups use "Tag category" for user-facing wording.
