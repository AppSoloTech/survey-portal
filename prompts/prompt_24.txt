# Received Feedback

After Phase 21, the product direction is that admins should be able to attach
hidden tags to the system-generated Other choice when Allow Other is enabled.
Admins do not need to edit the Other label, create Other as a normal answer
option, or use Other for conditional logic.

# Scope

Implement:

* A separate relational model for hidden tags attached to the system-generated
  Other choice, likely `question_other_tags`.
* Admin add/remove/edit UI for Other hidden tags only when:
  * question type is `single_select` or `multi_select`
  * `allowOther` is enabled
  * survey is an editable draft
* Tag catalog behavior consistent with normal answer-option hidden tags.
* Report/detail/CSV hidden-tag behavior:
  * when `other_text` is non-null, include the question's Other hidden tags
  * keep selected real option tags and Other tags distinct in the implementation
  * roll up Other tags in the existing hidden-tag summary
* Shared/admin response types for Other hidden tags as needed.
* Focused tests for storage, validation, reporting, CSV, and participant
  hidden-tag non-disclosure.

## Non-Goals

Do not implement:

* a normal `answer_options` row for Other
* editable Other label/copy
* conditional logic triggered by Other
* hidden tags based on matching specific Other text values
* Other support for `scale`, `text`, or `integer`
* aggregation or normalization of identical Other text values
* changes to Phase 20 admin-user/profile/contact tooling or `/api/profile`

## Assumptions

* Other hidden tags apply when any nonblank `other_text` is saved for the
  question.
* Other hidden tags are admin-only metadata and must never be sent to
  participant-facing survey APIs.
* Existing Phase 21 responses remain valid; adding tags later may affect future
  reports for historical Other responses because reporting resolves tags from
  current survey metadata.
* Other remains system-generated response behavior, not a normal answer option.

## Open Questions

* Should adding/removing Other hidden tags on a draft copy affect historical
  reports for the original published survey? If unanswered, follow the existing
  hidden-tag behavior for answer-option tags on the survey being reported.

---

# Acceptance Criteria

The phase is ready for review when:

* Admins can add/edit/remove hidden tags for Other on applicable draft questions
  with Allow Other enabled.
* Other hidden tags are not available on unsupported question types or when
  Allow Other is disabled.
* Participant APIs do not expose Other hidden tags.
* Admin attempt detail and CSV include Other hidden tags when `other_text` is
  present.
* Hidden-tag rollup counts Other tag events/respondents correctly.
* Other still is not represented as an answer option ID.
* Conditional logic remains unaffected by Other.

# Testing Requirements

Run:

```bash

```

Add focused tests for:

* creating/updating/deleting Other hidden tags
* rejecting Other hidden tags on unsupported question types
* rejecting Other hidden tags when `allowOther` is disabled
* tag catalog registration for Other hidden tags
* participant response payloads excluding Other hidden tags
* admin attempt detail showing Other hidden tags only when Other was answered
* hidden-tag rollup counting Other tags
* CSV hidden-tag output for Other responses
* ensuring Other remains absent from `selectedAnswerOptionIds`

Manual checks:

* Admin enables Allow Other, adds Other hidden tags, and saves.
* Participant submits an Other response.
* Admin Results shows the Other answer and the configured hidden tag.
* CSV includes the Other text and hidden tag.
* Participant runner never shows hidden tags.
* Turning Allow Other off hides or blocks Other hidden-tag editing as designed.
