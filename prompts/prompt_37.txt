# Received Feedback

1. The survey workspace tab bar clips on narrow screens without a clear scroll
   affordance.
2. The `/admin/tags` page should use "category" for higher-level tag catalog
   buckets instead of visible "group" wording.
3. The Ungrouped holding section should be movable like custom category groups.

Do not silently drop feedback. Anything not implemented must be moved to
Non-Goals, Deferred Work, or Open Questions.

---

# Intake Classification

* Bug or regression: clipped survey workspace tabs on narrow screens.
* UX polish: tag catalog wording should match the higher-level category concept.
* Feature request: persisted ordering for the Ungrouped catalog section.
* Database/schema change: yes, add a small persisted catalog setting.
* API contract change: yes, add an admin-only section reorder endpoint and
  include Ungrouped display order in the existing catalog response.
* Participant-facing survey behavior: no.
* Hidden-tag visibility: no participant exposure.
* Authentication or authorization: preserve admin-only tag catalog access.

---

# Scope

Implement:

* Make `.workspace-tabs` scrollable and discoverably overflowing on narrow
  screens without changing the existing route/tab structure.
* Update visible `/admin/tags` copy from "group" to "category" for the
  higher-level catalog buckets, including labels, buttons, helper copy,
  success/error/toast/confirm strings, and aria labels.
* Keep existing internal names such as `TagCatalogGroup`, `groupId`, and
  database `tag_groups` unless a visible API/type rename is unavoidable.
* Persist an ordered position for the Ungrouped catalog section.
* Add an admin-only catalog section reorder API that accepts exactly one
  Ungrouped virtual section plus every real tag category exactly once.
* Add response data needed by the frontend, such as `ungroupedDisplayOrder`,
  while preserving the existing `groups`, `tags`, and `ungroupedTags` response
  shape.
* Update drag logic so dragging a category or Ungrouped among sections persists
  section order; dragging tags still reorders and moves tags as today.
* Add/update focused API and pure drag tests.
* Update `markdown/releases/unreleased.md`, `markdown/PHASE_LOG.md`, and
  `notes/claude_handoff_phase_37.txt`.

## Non-Goals

Do not implement:

* Database table or shared type renames from group to category.
* Participant-facing survey payload changes.
* Hidden-tag meaning, tag key/value behavior, or survey builder tag editing
  changes.
* A broad admin workspace navigation redesign.

## Assumptions

* "Ungrouped should be movable just like custom category groups" means persisted
  ordering, not temporary frontend-only ordering.
* "Group" can remain an internal implementation term; visible admin copy changes
  to "category".

## Open Questions

* None blocking V1.

---

# Acceptance Criteria

The phase is ready for review when:

* On narrow screens, survey workspace tabs can be scrolled and visibly indicate
  more tabs are available.
* `/admin/tags` no longer shows user-facing "group" copy for the higher-level
  catalog buckets.
* Ungrouped can be dragged above, between, or below custom categories.
* Ungrouped position persists after reload.
* Tag dragging to and from Ungrouped still works.
* Existing category creation, rename, delete, tag move, and tag reorder workflows
  still work.
