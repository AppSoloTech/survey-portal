## Purpose

Refactor `/admin/glossary` into a tabbed Admin page and add the dynamic
question-search interface backed by the Phase 56 API.


This phase adds the UI for finding candidate Glossary terms in assessment
question text. It should not wire search results into Glossary entry creation

# Received Feedback

1. The Glossary page now needs to serve two distinct jobs: existing term
   management and question-text discovery.
2. The client wants a search bar that updates results dynamically as the Admin
   types.
3. Results should display below the search bar and refine with each added
   letter.
4. An Admin should be able to type an entire question and see the matching
   question emerge as a result.
5. This phase owns the search UI only; creation workflow wiring remains
   deferred to Phase 58.

Do not silently drop feedback. Anything not implemented must be moved to
Non-Goals, Deferred Work, or Open Questions.

# Scope

Implement:

* A tabbed layout on `/admin/glossary`:
  * one tab for existing Glossary entries
  * one tab for question search
* Preservation of all existing Glossary entry management behavior from Phases
  39 and 40.
* A web API helper for the Phase 56 question-search endpoint in
  `apps/web/src/api/glossary.ts`.
  * Endpoint: `GET /api/admin/glossary/question-search?q=<query>&limit=<optional>`.
  * Response: `{ query, minQueryLength, limit, results }`.
  * Result shape: `{ assessment, page, question, match }`, where
    `assessment` includes id/title/status, `page` is null or includes
    id/title/displayOrder, `question` includes id/questionText/displayOrder,
    and `match` includes start/end offsets.
  * Use the existing `apiRequest` `RequestInit` support for `AbortSignal`
    rather than changing the shared API client unless implementation inspection
    shows a better established pattern.
* A question search input that:
  * trims before search
  * waits for the Phase 56 minimum query length of 2 before calling the API
  * debounces requests, such as 250 ms
  * guards against stale responses by using `AbortController`, request ids, or
    an equivalent local pattern
  * uses a conservative request limit, such as the Phase 56 default of 20;
    do not add visible result-limit controls in this phase unless needed for
    usability
* Search states:
  * short/empty query
  * loading
  * results
  * no results
  * recoverable error
* Result display below the input with assessment/page/question context and
  highlighted matched text using the required match offsets from Phase 56.
  * Treat `match.start` and `match.end` as offsets into the returned original
    `question.questionText` string, not into a lowercased copy.
  * Preserve the original question text casing in rendered results.
* Accessible tabs and search results:
  * keyboard-operable tabs
  * visible focus states
  * useful labels
  * polite live-region updates for loading/result-count changes where fitting
* Responsive layout at common mobile, tablet, and desktop widths.
* A disabled or clearly non-creating result action placeholder if needed to
  prepare for Phase 58, without creating or pre-filling entries yet.

## Non-Goals

Do not implement:

* Add-to-Glossary creation wiring.
* Automatic Glossary entry creation.
* Duplicate detection in the create-entry workflow.
* Persisted source-question references.
* Backend endpoint changes beyond consuming the Phase 56 API.
* Search over answer options, hidden tags, response data, or non-question
  content.
* A new top-level Admin route unless the existing route cannot support the
  required accessible tab layout.
* Broad visual redesign of the Admin area.

## Assumptions

* Phase 56 has already shipped or is available on the branch.
* Phase 56 behavior is the source of truth for short queries and limits:
  blank or one-character queries return empty results; the API defaults limit
  to 20 and caps it at 50.
* The existing Glossary page can absorb tabs without losing the current CRUD
  workflow. The product direction is confirmed: keep question search as a tab
  inside `/admin/glossary`, not as a separate top-level Admin page.
* UI copy should use assessment terminology even if API/domain fields still use
  survey-oriented names internally.
* Search results are informational in this phase.
* Unicode case-folding edge cases called out in the Phase 56 review are accepted
  as non-blocking for this first English/plain-text Admin UI; highlight using
  the provided offsets without trying to recompute match locations client-side.

## Open Questions

* If the existing page structure makes accessible tabs unusually risky,
  document the proposed alternative before implementing it.

# Implementation Priorities

Handle items in this order unless the human explicitly changes priority:

1. Preserve existing Glossary entry management.
2. Build accessible tabs and search controls.
3. Implement responsive, stable result rendering.
4. Add request debounce and stale-response protection.
5. Cover the new UI behavior with focused tests where practical.

Keep changes narrowly scoped to this phase.

# Acceptance Criteria

The phase is ready for review when:

* `/admin/glossary` presents separate tabs for existing entries and question
  search.
* Existing Glossary create/edit/archive/dictionary behavior still works.
* Admins can type in the question search box and see dynamic results from the
  Phase 56 endpoint.
* Blank and too-short queries do not show stale results or fire broad searches.
* Loading, empty, error, and result states are visible and understandable.
* Result rows/cards include enough assessment/page/question context for an
  Admin to understand where the text came from.
* Result rows/cards highlight the matched substring; missing highlight behavior
  is not acceptable now that Phase 56 requires offsets.
* The search UI is keyboard-operable and screen-reader friendly.
* The layout remains usable on mobile, tablet, and desktop.
* No result action creates or pre-fills a Glossary entry yet.
* Every received feedback item is fixed, deferred with a reason, or blocked by a
  documented question.
