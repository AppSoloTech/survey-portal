# Received Feedback

1. The Glossary needs a second workflow that helps Admins find terms or phrases
   from existing assessment question text.
2. Search should cover all non-retired assessments, not only one assessment or
   one state.
3. Search should refine dynamically as the Admin types, eventually allowing a
   full question to appear when the full question text is typed.
4. This phase owns only the API foundation needed for that later UI.

Do not silently drop feedback. Anything not implemented must be moved to
Non-Goals, Deferred Work, or Open Questions.

# Scope

Implement:

* Shared response types for Admin Glossary question search in
  `packages/shared/src/index.ts`.
* An Admin-only endpoint under the existing Glossary Admin API surface, such as
  `GET /api/admin/glossary/question-search?q=...`.
* Query validation:
  * trim whitespace
  * reject or return an empty result for queries shorter than the chosen minimum
    length, with consistent API behavior documented in tests
  * cap result count with a conservative default, such as 20, and a hard upper
    bound, such as 50
* Question-text-only search over assessment questions.
* Filtering that includes draft and published assessments while excluding
  retired assessments and soft-deleted records.
* Case-insensitive substring matching. Do not implement fuzzy search yet.
* Deterministic result shape including enough context for the UI:
  * assessment id, title, and status
  * page id, title, and display order when available
  * question id, question text, and display order
  * required match start/end offsets for the first case-insensitive substring
    match in `questionText`
* Deterministic ordering suitable for repeated keystroke searches:
  * earliest match position first
  * then assessment title
  * then page display order
  * then question display order
  * then question id as a final tie-breaker
* Focused API tests for authentication, role authorization, validation, status
  filtering, soft-delete filtering where the schema supports it, and returned
  result shape.

## Non-Goals

Do not implement:

* Admin Glossary tabs or any UI changes.
* Search result rendering, debounce, loading states, or empty states.
* Add-to-Glossary creation wiring.
* Persisted source-question references on Glossary entries.
* Database schema changes, migrations, indexes, or full-text-search
  infrastructure unless the existing schema already requires a tiny supporting
  adjustment to make tests pass.
* Search over answer options, page descriptions, assessment descriptions,
  hidden tags, reports, or response data.
* Fuzzy matching, stemming, ranking engines, or external search providers.
* Participant-facing API changes.

## Assumptions

* Developer-facing route, table, and type names may continue to use
  survey-oriented identifiers where the current codebase does.
* User-facing strings introduced later should say assessment, but this API phase
  may return fields aligned with existing domain types.
* Non-retired means draft and published assessment records only.
* The first implementation can use database substring matching because this is
  an Admin helper workflow, not a public high-volume endpoint.
* Empty or short queries should not scan all questions.
* Leading-wildcard PostgreSQL `ILIKE` is acceptable for the first Admin helper
  version, but should be revisited if real content volume approaches or exceeds
  roughly 10k questions.

## Open Questions

* None blocking Phase 56. Future phases may revisit full-text search,
  `pg_trgm`, or another indexed search approach if real assessment volume makes
  substring search too slow.

# Acceptance Criteria

The phase is ready for review when:

* Non-admin and unauthenticated users cannot access the question-search
  endpoint.
* Admins can search question text across draft and published assessments.
* Retired assessments are excluded.
* Hidden tags, response data, answer data, and Admin-only glossary source
  metadata are not returned.
* Short or blank queries do not trigger broad unbounded searches.
* Returned results include assessment/page/question context needed by the
  planned UI.
* Returned results include match start/end offsets for highlighting.
* Results follow the documented deterministic ordering.
* API tests cover the new behavior and relevant edge cases.
* Every received feedback item is fixed, deferred with a reason, or blocked by a
  documented question.
