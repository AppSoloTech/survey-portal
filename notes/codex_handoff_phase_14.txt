What changed
------------
apps/api/src/services/surveyAttempts.ts
  - New pruneOffPathAnswers(queryable, survey, attemptId): recomputes the
    revealed-question set from the attempt's CURRENT answers via
    resolveProgressivePageState, then
        delete from survey_response_answers
        where survey_attempt_id = $1 and question_id <> all($2::int[])
    with the keep-set = union of visibleQuestionIdsByPageId.
    survey_response_selected_options rows cascade
    (database/migrations/0003_surveys.sql on delete cascade).
    Guards: returns early (no delete) when hasLoop, or when the keep-set is empty.
  - New fetchAttemptResponses(queryable, attemptId): transaction-scoped loader
    (mirrors fetchAttemptsByCondition's response/selected-option hydration but on
    the supplied queryable) so the prune sees the just-saved rows and the
    selected options that jump rules key off.

apps/api/src/routes/surveyAttemptRoutes.ts
  - POST /:id/answer: after saveAnswer, fetch the survey structure
    (fetchSurveyStructures, includeAllStatuses:true, includeHiddenTags:false) and
    call pruneOffPathAnswers — all still inside the existing transaction, before
    commit.
  - POST /:id/pages/:pageId/answer: after savePageAnswers (survey already
    fetched), call pruneOffPathAnswers once per request, before commit.
  - /complete intentionally NOT touched: it has no transaction wrapper, and every
    answer/page save already prunes, so data is clean by completion.

KEY DESIGN POINT to scrutinize — keep-set source
-------------------------------------------------
The keep-set comes from resolveProgressivePageState (the PAGE runtime's resolver),
NOT the question-level resolveAttemptPath used by surveyReporting.ts.
Reason: resolveAttemptPath does not reroute on JUMP_TO_PAGE the way the page
runtime does. An intermediate version pruned off resolveAttemptPath and the
page-based repro test FAILED (Engineering answer survived); switching to
resolveProgressivePageState fixed it. resolveProgressivePageState handles
JUMP_TO_QUESTION, JUMP_TO_PAGE, and HIDE_QUESTION, so the prune cannot diverge
from what the respondent was shown.

Note the slight asymmetry worth a look: the prune uses resolveProgressivePageState
while collectFinalPathQuestionIds (the onFinalPath badge / safety net) still uses
resolveAttemptPath. After pruning, answered-off-path rows shouldn't exist so the
badge is vestigial for answered rows; confirm the two cannot disagree in a way
that mislabels a surviving row.

Suggested review focus
----------------------
1. Data-loss safety: confirm the delete can only ever remove genuinely off-path
   rows. Not-yet-reached questions have no rows; the keep-set is the union of
   revealed/projected question ids; empty keep-set and hasLoop both skip the
   delete. Check the `<> all($2::int[])` semantics with the chosen guard (an empty
   array would otherwise match all rows — guarded by the length===0 return).
2. Transaction boundaries: both call sites run before commit, so a failed prune
   rolls back the save. fetchAttemptResponses reads via the same client (sees the
   just-saved rows); fetchSurveyStructures reads survey structure via the pool
   (unchanged within the txn) — confirm that's acceptable.
3. Keep-set correctness vs the runtime: resolveProgressivePageState is the right
   single source of truth (see KEY DESIGN POINT). Confirm in-progress projection
   can't false-delete and that the onFinalPath/resolveAttemptPath asymmetry is
   harmless.
4. Scope = all off-path: verify HIDE_QUESTION-hidden answers really should be
   pruned (developer chose yes). reporting.test.ts:317 now asserts deletion.
5. Performance: one extra SELECT(+options) and one DELETE per answer/page save,
   plus a survey-structure fetch added to the single-answer route. Flag if that
   per-keystroke cost is a concern for the single-answer flow.
