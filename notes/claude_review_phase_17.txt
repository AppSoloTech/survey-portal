## 1. Critical / blocking issues

None that block commit outright. The closest thing to a blocker is a design decision rather
than a defect:

- public_token stores the FULL working token (`asl.<lookupKey>.<secret>`) in plaintext, in the
  same row as token_secret_hash. This means the SHA-256 hash now provides ZERO additional
  protection: anyone who can read the row (DB compromise, read-only SQL injection, an
  unencrypted backup, broad DB credentials) obtains live, working survey links. The previous
  design's security property ("the token cannot be reconstructed after creation") has been
  traded away to enable repeat-copy.

  Why I rate this important-but-not-blocking: an anonymous survey link is a low-sensitivity
  capability URL. It grants exactly one ability — submit an anonymous response to a published
  survey — and that ability is now rate-limited. It does not expose PII, existing responses,
  hidden tags, or any admin/auth surface. The links are designed to be widely shared (emailed,
  posted), so their confidentiality requirement is inherently low. Given that, plaintext storage
  is defensible. But the team should decide deliberately (section 5), document it, and stop
  carrying a hash that implies protection it no longer provides.

Everything else in the requested focus list checks out (details in sections 2/4).

## 2. Non-blocking security / privacy concerns

a) Redundant hash alongside plaintext token (maintainability + honesty).
   With public_token stored, token_secret_hash + timingSafeEqual is vestigial. Keep ONE model:
   either drop the hash and match on the stored token, or (preferred) encrypt public_token and
   keep the hash for lookup verification. Carrying both invites a future reader to assume the
   tokens are protected at rest when they are not.

b) In-memory rate-limit store does not span instances.
   `new MemoryStore()` is per-process. On Azure App Service with >1 instance (or after a
   restart), the effective limit is N × max and counters reset on deploy. Fine for a single
   instance / MVP, but note it: a shared store (Redis/Postgres) is needed before horizontal
   scaling for the limit to be real. Also, IP-keying means participants behind a shared
   corporate NAT share a bucket — 120/15min is generous enough that false positives are
   unlikely, but worth knowing.

c) Anonymous follow-up email is unverified PII surfaced to admins.
   `anonymous_contact_email` is free-typed, never verified, and rendered in admin Results and
   CSV through `ReportParticipant.email` (disambiguated only by `type: "anonymous"`). Two notes:
   - Treat it as PII in the data model docs; it partially de-anonymizes an otherwise anonymous
     attempt by design (opt-in), so it should be called out as such.
   - In the admin UI/CSV, make sure it is labeled as an unverified, self-supplied follow-up
     address, not mistaken for a verified account email. The CSV currently reuses the same
     participant email column for both kinds.
   The capture flow itself is correctly gated: post-completion only (409 before), owner-scoped
   by link id + per-attempt token hash + survey id, length/shape validated, lowercased, and
   constrained at the DB layer to anonymous attempts only. Good.

d) Token still travels in the URL path.
   Mitigated for app logs (request-logger masks `/api/anonymous-surveys/[token]`) and for
   referrers (no-referrer app-wide). Residual exposure is the participant's own browser history
   and any intermediary that logs full URLs outside our app — inherent to shareable-link
   designs; acceptable, just not zero.

## 3. UX / process suggestions

a) Contact-email modal over-promises. The copy reads "Want survey result updates? Enter an
   email if you would like more info regarding the results of the survey." There is no email
   delivery in the product yet (Phase 16 shipped a disabled/no-op foundation), so today this
   email is only stored and shown to the admin — no updates can be sent. Either soften the copy
   to reflect that the survey owner may follow up, or hold the result-updates promise until a
   real provider exists. Also consider adding a short "the survey owner will be able to see this
   address" disclosure for informed consent, since it de-anonymizes.

b) Repeat-copy is admin-only and correctly hides the URL for disabled links (test covers both),
   which is the right behavior. Consider showing a subtle "treat this link as shareable but not
   secret" hint near the copy button so admins don't assume it is a private credential.

c) Document the public_token decision in DATA_MODEL_VISION.md (sensitivity classification,
   at-rest handling) so the next phase doesn't reintroduce a hashed-only assumption.

## 5. Specific recommendation on public_token: ACCEPT (with hardening) — do not remove

Decision among accept / encrypt / remove / redesign:

- REMOVE is wrong for this use case. A survey link is distributed once and widely; admins need a
  stable, retrievable URL to re-copy. Reverting to copy-once breaks the legitimate workflow that
  motivated this change, and rotating the token on each copy would invalidate links already
  emailed out.
- ACCEPT plaintext is reasonable on its own merits: the token is a low-sensitivity capability
  URL (submit-only, rate-limited, no PII/admin reach), exposed via API to admins only.
- ENCRYPT is the recommended hardening if you want defense-in-depth: store public_token
  encrypted at rest (AES-GCM with an app/Key-Vault-held key), keep token_secret_hash for the
  fast lookup-key verification, and decrypt only when an admin requests the URL. This preserves
  repeat-copy while ensuring a DB-only or backup-only compromise does not yield live links.

Recommendation: ACCEPT for now as an explicitly documented, low-sensitivity decision, AND
reconcile the redundant hash. Concretely:
  1. Add a sensitivity note to DATA_MODEL_VISION.md: anonymous link tokens are shareable
     capability URLs, stored retrievably to support re-copy, protected primarily by DB access
     control and rate limiting.
  2. Either drop token_secret_hash (and match on the stored token) OR — preferred if survey
     participation is considered sensitive in your context — encrypt public_token at rest and
     keep the hash for verification.
  3. Ensure DB backups and connection credentials are access-controlled (they should be already).
Encryption can land in a small follow-up; it does not need to block this commit.

What resolved my prior findings:

- public_token is now AES-256-GCM ENCRYPTED at rest (format enc:v1:<iv>:<tag>:<ct>,
  per-record random 12-byte IV, GCM auth tag). token_secret_hash is retained for the
  timing-safe lookup verification, and the plaintext is decrypted only when an admin
  reads the URL. This is exactly the "encrypt at rest, keep the hash for lookup"
  option I recommended in section 5 — it closes the section-1 concern: a DB-only or
  backup-only compromise no longer yields live links without also holding the app
  encryption secret. Migration 0019 widens the CHECK to allow enc:v1:% (still permits
  legacy asl.% for back-compat). Test asserts the stored value is enc:v1:* and does
  not contain the plaintext token. RESOLVED.
- Rate-limit store moved off in-memory to a shared Postgres table
  (PostgresRateLimitStore, migration 0020), so the limit is now real across instances
  and restarts. Bonus privacy: it stores SHA-256(client key), not raw IPs. Closes
  prior finding 2b. RESOLVED.
- Link rotation + disabled-link cleanup: POST /:id/anonymous-links/:linkId/rotate
  (admin-only) transactionally issues a fresh token and disables the old one
  (SELECT ... FOR UPDATE, rollback on failure, re-checks survey still published).
  Test confirms the old token becomes unavailable and the new one works. Sound.

New minor (non-blocking) notes from this delta:

a) Encryption key reuse. anonymousLinkTokenEncryptionSecret falls back to the JWT
   secret when ANONYMOUS_LINK_TOKEN_ENCRYPTION_SECRET is unset. Reusing one secret for
   two cryptographic purposes (HMAC-JWT and AES-GCM) is a crypto-hygiene smell rather
   than a vulnerability — the key is derived via SHA-256 and stays server-side, and in
   prod it inherits the JWT secret's length/placeholder guards. Recommend setting a
   dedicated secret in prod (the env doc already marks it "optional"); consider a prod
   guard so it is not silently shared.
b) Rate-limit pruning cost. PostgresRateLimitStore.increment() runs a global
   DELETE ... WHERE window_reset_at <= now() on EVERY anonymous request. With the
   window_reset_at index this is a ranged delete, but on a hot public endpoint it is
   write amplification plus dead-tuple churn and possible contention. Prefer
   probabilistic pruning (e.g. ~1% of calls) or a scheduled sweep; the get/increment
   window checks already ignore expired rows, so eager pruning is not required for
   correctness.
c) Key-rotation behavior (document it). If the encryption secret changes, existing
   rows fail to decrypt and decryptPublicToken returns null, so admin re-copy/list
   simply omits publicUrl — the links themselves still work for takers (verification
   uses lookup key + secret hash, independent of public_token). Graceful, but worth a
   one-line note in DATA_MODEL_VISION so a future secret rotation is not mistaken for
   data loss. The legacy asl.% passthrough in decrypt is harmless for this unreleased
   phase (no released plaintext rows to migrate).

Verdict: COMMIT-READY. The two items I had carried as recommended hardening
(encrypt public_token, shared rate-limit store) are now done, and the rotate flow is
a clean addition. Remaining items (a/b/c above, plus the earlier contact-email modal
copy and the unverified-email labeling in Results/CSV) are all non-blocking polish.
