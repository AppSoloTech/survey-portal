## Findings / Follow-ups (non-blocking)

1. Read-time redaction is heuristic; the durable guarantee belongs at write time.
   sanitizeOperationalString has known gaps the later suite runner must not rely on:
   - camelCase secrets in free-text strings escape the key=value regex because of \b
     word boundaries (e.g. "accessToken=x", "apiKey=x" are only caught as JSON object
     keys, not inside string values).
   - "api_key"/"apikey" is absent from both the string regex and isSensitiveOperationalKey;
     recommend adding it.
   - A secret serialized as JSON *inside* a string value (e.g. '{"password":"x"}') is
     not matched because the quote between key and ':' blocks the key=value regex.
   Acceptable as defense-in-depth for Phase 52. The Phase 53+ suite runner should avoid
   persisting secrets at write time rather than leaning on this read-time scrub.

2. Consistency gap: GET /api/admin/performance-runs/:id (Phase 49-51) still returns raw
   config/summary/reportMarkdown and an unsanitized targetBaseUrl. This phase added
   redaction to suites/samples but the sibling run-detail endpoint exposes the same
   class of operational payload unredacted. Recommend a follow-up to extract the
   sanitizer into a shared util and apply it to run detail too.

3. Suite-detail child-runs query has no LIMIT. Fine in practice (capacity suites have a
   small bounded profile set), but it is the one read in the detail path without a cap.
   Consider a sane cap or an explicit documented assumption.

4. Samples expose limit only — no offset/cursor — so a future viewer cannot page past
   the 1000-row cap for a long suite. Within prompt scope ("bounded window"); flag for
   the viewer phase.

5. Minor: readSampleQuery accepts an undocumented "limit" alias (query.sampleLimit ??
   query.limit). Harmless; either document it or drop it. The redactor also rewrites ':'
   to '=' in matched pairs (cosmetic).
