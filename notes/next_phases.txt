4) Get more user info via profile to collect additional user demogrphics. User profile should contain user metadata as well as some user survey statistics.
5) Increase the amount of base user related functionality the admin has (view user profile info, initiate a password rest...).
6) For choice-based questions, the admin can add an “Other” option that lets the survey taker type their own answer. Add support for an optional “Other” answer choice on applicable question types. For multiple-choice, checkbox, dropdown, and similar option-based questions, admins should be able to enable an “Allow Other” setting. When enabled, the survey-taking UI should automatically render an “Other” option with a free-text input field for the respondent’s custom answer.
The “Other” option should be system-generated and not treated as a normal manually entered answer option. Store the selected Other response separately from standard option IDs, including the custom text entered by the respondent.
7) Definitions tooltip for questions. Add support for optional question definitions/help text. Survey administrators should be able to enter explanatory text for each question. If provided, the survey-taking interface should display an information icon adjacent to the question text. Selecting the icon should display the definition/help text in a tooltip or popover without navigating away from the survey. The definition is informational only and should not affect survey logic or reporting.
8) On the survey question page, replace numeric progress indicators with an estimated time remaining display.The survey-taking interface should no longer show “Question X of Y” or “Page X of Y” labels. Instead, display an estimated time remaining value based on the number and type of remaining questions/pages. The estimate can be calculated using a configurable average time per question, with optional adjustments by question type.
9) Follow-up to the Phase 21 “Other” answer foundation: allow admins to attach hidden tags to the system-generated Other choice only. Keep Other out of normal answer_options, do not add editable Other labels, and do not make Other trigger conditional logic. When a participant provides Other text, reports and CSV should resolve the configured Other hidden tags while participant-facing APIs continue to hide all tag metadata.
