Source of the request
---------------------
Developer: "On the logic page, I would like to include a new 'Skip page' option
that is similar to the 'Skip questions' use case. I would like users to have the
option to skip entire pages conditional to the results of the Source question's
answer."

Decisions settled before implementing (confirmed with the requester)
--------------------------------------------------------------------
1. Source parity with Skip questions: BOTH selection sources (equals) AND text
   "answer is blank" sources (is_blank) can drive a Skip page rule.
2. Skip scope: WHOLE PAGE, DYNAMIC — hide every question currently on the target
   page at evaluation time (so a question added to that page later is also
   skipped). Chosen over snapshotting questions at create time.

Core design — why this is small and low-risk
--------------------------------------------
New first-class action type HIDE_PAGE (target_page_id set, target_question_id
null, skip_target_in_normal_flow forced false — exactly like HIDE_QUESTION but
page-targeted, paralleling JUMP_TO_PAGE vs JUMP_TO_QUESTION).

At runtime a HIDE_PAGE rule is reduced to "all questions on the target page are
hidden": the ONLY runtime change is in
packages/shared/src/index.ts > collectActivatedHiddenQuestionIds — for a matching
HIDE_PAGE rule it pushes getQuestionsForPage(survey, targetPageId) ids into the
SAME activeHiddenQuestionIds set that HIDE_QUESTION already feeds. Because both
path walkers (resolveProgressivePageState, resolveAttemptPath), the reporting
keep-set (collectFinalPathQuestionIds), and pruneOffPathAnswers all key off that
hidden-question set, page-skip, reporting "not_reached", and off-path pruning all
work with NO further change. getNextVisiblePage already advances past a page once
all its questions are hidden.

KEY POINT to scrutinize: an empty target page contributes zero hidden ids. That
is fine — an empty page already renders nothing and getNextVisiblePage skips it in
normal flow — but confirm you agree there's no edge case where an admin expects an
empty page to be "marked skipped".

Addendum — "Advance on trigger" lever for Skip page (same session)
------------------------------------------------------------------
Request: the page-skip only takes effect after the trigger question's whole page
is completed. The developer asked for an admin-controlled lever to instead skip
the rest of the source page and jump past the skipped page(s) the moment the
trigger answer is submitted.

Design: new per-rule boolean `advance_on_trigger` (HIDE_PAGE only; forced false
elsewhere). Default false preserves the prior "finish the page, then skip"
behavior. When true and the trigger fires, the page walker stops revealing the
current page and advances to getNextVisiblePage (which skips the just-hidden
target page) — the same immediate-navigation path JUMP rules already use.

Changes:
- DB: database/migrations/0015_hide_page_advance_on_trigger.sql (column + check
  constraint: only HIDE_PAGE may set it).
- Shared: ConditionalLogicRule.advanceOnTrigger; new helper
  hasActivatedAdvancingPageSkip; resolveProgressivePageState advances when it
  returns true (after the JUMP check, before continuing to the next same-page
  question).
- API: surveyRecords (record + mapper + select), surveyStructure (select),
  surveyBuilder duplication (type + select + insert), routes (insert/update
  $10/$11), validation (parse + force false unless HIDE_PAGE).
- Web: api/surveys.ts (create/update bodies), RuleCreateForm (checkbox under
  "Pages to skip", applies to all pages in the batch), RuleEditor (checkbox in
  the flow-toggle cell for HIDE_PAGE), RuleList (persist on save), flow map
  (edge field + "Advances immediately..." note).
- Tests: shared (advance vs. finish-page), API (validation force-off + e2e:
  AfterRoute never revealed, lands on Final), web flow-graph makeRule default.

Validation: typecheck/lint green; shared 47, web 52, api 157; migrations 0014 +
0015 applied by the API test globalSetup.

Scrutinize: advance is OR-ed across a source's HIDE_PAGE rules (any advancing
rule that matches on the answered question advances). The remaining same-page
questions after the trigger become off-path and are pruned, exactly like a jump.
