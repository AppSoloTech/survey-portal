## Phase Scope

Implemented:

- Added admin-only survey builder APIs under `/api/surveys`.
- Added `PATCH /api/surveys/:id/status` for draft, published, and retired transitions.
- Tightened legacy survey metadata update behavior so publishing validates survey readiness.
- Prevented creating a survey directly as `published`.
- Added admin-only question management:
  - create questions
  - edit question text
  - edit help text
  - edit required flag
  - edit question type while the survey is still draft
  - delete questions
  - reorder questions without tripping display-order unique constraints
- Added admin-only answer option management for selection questions:
  - create options
  - edit option text
  - delete options
  - reorder options without tripping display-order unique constraints
- Added admin-only hidden answer tag management:
  - create tags
  - edit tags
  - delete tags
  - handle duplicate tag key/value conflicts
- Added admin-only conditional rule management for MVP jump rules:
  - create `JUMP_TO_QUESTION` rules
  - edit rules
  - delete rules
  - force `conditionOperator = equals`
  - force `actionType = JUMP_TO_QUESTION`
- Added server-side validation for:
  - survey title required and length-bounded
  - survey description length-bounded
  - survey status values
  - question text required and length-bounded
  - question help text length-bounded
  - question type values
  - answer option text required and length-bounded
  - answer tag key/value required and length-bounded
  - positive display order and resource ids
  - nested parent ownership for surveys, questions, options, tags, and rules
  - conditional rule references all belonging to the same survey
  - source answer option belonging to the source question
  - source question being `single_select` or `multi_select`
- Added publish validation requiring:
  - at least one question
  - every selection question has at least one answer option
  - conditional rule references are valid and limited to `equals`/`JUMP_TO_QUESTION`
- Preserved Phase 3 standard user behavior:
  - users can only start published surveys
  - already-started retired survey attempts remain resumable/completable
  - draft surveys remain inaccessible to standard users
  - standard user survey APIs still call survey structure loading with hidden tags disabled
- Added admin frontend API helpers for survey builder writes.
- Replaced the placeholder admin dashboard with a form-based builder UI.
- Admin UI now supports:
  - survey creation
  - survey metadata editing
  - publish/retire controls
  - question creation/editing/deletion/reordering
  - answer option creation/editing/deletion/reordering
  - hidden tag creation/editing/removal
  - conditional jump rule creation/editing/deletion
- Added responsive CSS for a dense operational admin builder.

Non-goals:

- Drag-and-drop builder.
- Analytics.
- Reporting.
- CSV export.
- Advanced visual builder.
- Scoring engine.
- Additional conditional action types:
  - `SHOW_QUESTION`
  - `HIDE_QUESTION`
  - `JUMP_TO_PAGE`
  - `END_SURVEY`
- Rule evaluation changes beyond preserving existing Phase 3 `JUMP_TO_QUESTION` behavior.
- Database schema changes.
- Automated route tests.
- Browser automation or screenshot review.

Known tradeoffs:

- `apps/api/src/routes/surveys.ts` is now very large. This kept Phase 4 close to the existing survey structure mapping and Phase 3 attempt logic, but the route should likely be split into focused service/router modules before reporting or tests add more behavior.
- The admin UI is intentionally form-based and operational, but it is a large single component. It may deserve extraction into smaller builder components after review.
- Delete operations rely on the existing explicit foreign keys and cascade behavior. This is aligned with the schema, but review should confirm there are no response-history implications that should block deletion for published surveys.
- The UI lets admins edit metadata/status through the metadata form and through publish/retire buttons. The backend is the source of truth and applies publish validation on both paths.
- Reordering uses temporary positive display-order offsets before assigning final contiguous order values to avoid unique constraint conflicts.
- Existing Phase 3 user route code remains in the same file as the new builder routes, so review should be careful to distinguish intended admin behavior from user survey-taking behavior.

Deferred work:

- Add automated backend tests for admin authorization, publish validation, nested ownership validation, and hidden tag exposure.
- Add route-level tests for invalid cross-survey conditional rules.
- Add browser-based layout inspection for `/admin`.
- Extract survey builder route/service helpers from `apps/api/src/routes/surveys.ts` after the Phase 4 behavior is reviewed.
- Consider published-survey editing policy before production use, especially whether deleting questions/options from published surveys should be blocked or versioned.
- Update `markdown/PHASE_LOG.md` after Claude review and any review fixes are complete.
