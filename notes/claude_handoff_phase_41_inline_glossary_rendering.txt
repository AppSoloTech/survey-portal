Phase 41 - Participant Inline Glossary Rendering

Prompt:
prompts/prompt_41.txt

Branch:

Implementation summary:
- Added shared glossary text matching in packages/shared/src/index.ts:
  buildGlossaryTextSegments handles no matches, repeated matches,
  case-insensitive matching with original casing preserved, aliases/canonical
  terms, longest-match-first overlaps, and word-boundary checks.
- Added focused matcher tests in packages/shared/test/glossaryText.test.ts.
- Added apps/api/src/services/glossary.ts to centralize the participant-safe
  glossary projection.
- Reused that projection from the existing Admin-gated
  /api/admin/glossary/participant-safe route.
- Added glossaryEntries to authenticated runner start/resume payloads and
  anonymous runner start payloads. The payload contains only id,
  canonicalTerm, definition, and matchStrings.
- Added InlineGlossaryText for participant/admin-preview prompt rendering with
  hover, focus, click/tap, Escape, aria-expanded, aria-describedby, and tooltip
  role behavior.
- Styled glossary terms with a dotted underline plus subtle hover/focus
  highlighting so defined terms are discoverable before interaction.
- Rendered glossary matches in participant question prompts and Admin preview
  question prompts only.
- Added API tests for authenticated and anonymous runner payload isolation,
  including disabled-entry exclusion and absence of definitionSource,
  sourceProvider, sourceReference, and sourceLookupAt.
- Updated release notes, phase log, and follow-up backlog.

Key implementation decisions:
- Participant exposure is attempt-scoped: the participant UI does not call the
  Admin-only glossary route. Existing authenticated/anonymous authorization
  checks happen before glossary entries are included in runner payloads.
- Glossary rendering is prompt-only for Phase 41. Page descriptions, question
  help text, and answer option labels remain a follow-up pending client
  confirmation.
- The matcher is plain-text only. It does not parse HTML, does not treat match
  strings as regular expressions, and does not render nested/overlapping
  glossary spans.
- The visible question prompt is now the fieldset legend in the participant
  runner so glossary buttons are not placed inside aria-hidden content.
