## Phase Scope

Implemented:

- Added `database/migrations/0021_password_reset_tokens.sql` with `user_id`, public lookup key, hashed secret, expiry, consumed timestamp, and timestamps.
- Added `apps/api/src/services/passwordReset.ts` for `prt.lookup.secret` generation, SHA-256 secret hashing, timing-safe comparison, reset URL generation from `WEB_ORIGIN`, email dispatch through the Phase 16 email client, and transactional completion.
- Added public reset request and completion endpoints:
  - `POST /api/auth/password-reset/request`
  - `POST /api/auth/password-reset/complete`
- Added authenticated reset initiation endpoint:
  - `POST /api/auth/me/password-reset/request`
- Reused registration password validation rules for reset completion.
- Added separate in-memory auth-style rate limiters for public reset request and completion endpoints.
- Kept public request responses generic for existing, unknown, and syntactically invalid emails.
- Used URL fragments for reset links: `/reset-password#token=...`, so reset secrets are not sent to the server in normal browser navigation.
- Added request-log sanitization for accidental `/reset-password?token=...` navigations.
- Added frontend:
  - login "Forgot password?" link
  - `/forgot-password`
  - `/reset-password`
  - `/settings` protected account/settings page with logged-in reset initiation
- Added focused API tests for enumeration resistance, token storage, email payload dispatch, single-use reset, old password rejection, expired/malformed/unknown token failure, auth requirement, rate limiting, and no unexpected cookies.
- Updated `markdown/GLOBAL_DEVELOPMENT_ENVIRONMENT.txt` to clarify that `WEB_ORIGIN` is used for password reset links.

Non-goals respected:

- No admin-triggered reset workflow was added.
- No real email provider adapter was added.
- No plaintext reset token column was added.
- No encrypted/recoverable reset-token column was added.
- Existing sessions are not invalidated after reset.
- No unrelated profile redesign was added beyond the small `/settings` reset entry point.

Known tradeoffs:

- Existing authenticated sessions remain valid after a password reset because there is no token/session revocation model yet.
- Reset request/complete rate limiting uses the existing auth in-memory pattern, not the Phase 17 PostgreSQL-backed store. This matches the prompt preference and the existing `Auth And Security` follow-up about horizontal scaling.
- With email disabled in production, reset tokens can be generated but no real mail is delivered until a provider adapter is approved and implemented.
- Successful reset consumes all outstanding reset tokens for the same account, not only the used token.

Deferred work:

- Real production email delivery remains deferred to the existing future email-provider follow-up.
- Shared PostgreSQL-backed auth/reset rate limiting remains deferred with the existing horizontal-scaling follow-up.
- Existing session invalidation after password reset should be revisited if/when the app gains session/version revocation.
