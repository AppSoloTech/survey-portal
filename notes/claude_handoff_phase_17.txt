## Phase Scope

Implemented:

- Added `database/migrations/0016_anonymous_survey_links.sql`.
- Added admin-created anonymous survey links scoped to published, non-deleted surveys.
- Link URL token format uses a public lookup key plus high-entropy secret.
- Database stores only the lookup key and SHA-256 hash of the token secret.
- Added nullable `survey_attempts.user_id`, `anonymous_link_id`, and `anonymous_access_token_hash`.
- Added a DB owner check so attempts are owned by either a registered user or an anonymous link plus attempt token hash.
- Added per-attempt anonymous access tokens for public answer/complete calls.
- Added admin routes:
  - `GET /api/surveys/:id/anonymous-links`
  - `POST /api/surveys/:id/anonymous-links`
  - `PATCH /api/surveys/:id/anonymous-links/:linkId/disable`
- Added public routes:
  - `GET /api/anonymous-surveys/:token`
  - `POST /api/anonymous-surveys/:token/start`
  - `POST /api/anonymous-surveys/:token/answer`
  - `POST /api/anonymous-surveys/:token/pages/:pageId/answer`
  - `POST /api/anonymous-surveys/:token/complete`
- Public routes revalidate link token, enabled state, expiration, survey status, and soft-delete state.
- Public routes reuse existing answer validation, page save, off-path pruning, completion validation, and participant-safe survey structure loading.
- Request logging masks anonymous survey token path segments.
- Admin reporting and CSV export label anonymous attempts without synthetic users.
- Admin Setup page can create, copy-on-create, list, and disable anonymous links.
- Added public frontend route `/anonymous-surveys/:token` reusing the participant runner without auth.
- Updated data-model docs, follow-ups, and phase log.
- Added focused API tests for admin-only endpoints, token unavailability, lifecycle, hidden-tag isolation, owner separation, and reporting representation.

Non-goals respected:

- No anonymous user accounts.
- No public survey discovery page.
- No `/api/my-surveys`, dashboard, profile, admin, hidden-tag, or auth-only access for anonymous users.
- No email invitation delivery.
- No recipient list management.
- No assignment rules.
- No password reset behavior.

Known tradeoffs:

- The full public URL can be shown/copied only immediately after creation. Because only the token secret hash is stored, the full token cannot be reconstructed after reload.
- Anonymous attempts do not yet have a refresh/resume mechanism. Reloading the public route starts a new anonymous attempt.
- Disabling or expiring an anonymous link blocks in-progress anonymous writes for that token.
- Link secret hashing uses SHA-256 over a high-entropy random secret rather than a slow password hash; this is intended for random bearer tokens, not human passwords.

Deferred work:

- Decide whether anonymous attempts need browser refresh/resume through session storage or another anonymous session mechanism.
- Future invitation/email phases may send anonymous links once a real email provider exists.
- Recipient-list management, invite tracking, opt-in/out handling, and assignment workflows remain out of scope.

## Specific Questions

- Does any public anonymous route leak hidden tags, value tags, admin metadata, auth-only data, or user-dashboard access?
- Is the token model acceptable: lookup key + hashed link secret, plus per-attempt hashed access token?
- Are anonymous attempt ownership checks strong enough for answer/page-answer/complete writes?
- Should disabling/expiring a link block already-started anonymous attempts, or should started attempts be allowed to finish?
- Is request-log token masking sufficient for `/api/anonymous-surveys/:token/*` paths?
- Does reporting safely represent anonymous attempts without inventing user accounts?
- Did the phase accidentally implement email invitations, recipient management, public discovery, or account creation?
