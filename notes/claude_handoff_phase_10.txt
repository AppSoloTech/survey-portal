2. Contract changes
-----------------------------------------------------------------------

New endpoints (all admin-only unless noted):
- GET /api/admin/users?page&pageSize, PATCH /api/admin/users/:id/role
- DELETE /api/surveys/:id, POST /api/surveys/:id/duplicate
- GET/POST/PUT/DELETE /api/categories[/:id]
- GET/POST/PUT/DELETE /api/tags[/:id]

Changed payloads:
- Survey gained categoryId, categoryName, deletedAt (additive). Survey
  create/update bodies accept optional categoryId.
- Shared types added: SurveyCategory(+Responses), TagDefinition(+Responses),
  AdminUserSummary, AdminUsersListResponse, AdminUserRoleResponse,
  AttemptPathResult, resolveAttemptPath.

Behavior changes:
- POST /api/surveys/:id/start now checks the survey row first; deleted
  surveys 404 on both resume and fresh-start paths (previously a deleted
  survey could not exist). Non-published handling unchanged.
- answer/complete return 409 for deleted surveys before attempt checks.

Constraint preservation (verified by tests):
- Error shape { "error": string } everywhere, including the new routers.
- Hidden tags: catalog and tag routes are requireRole("admin");
  participant payloads asserted tag-free via collectObjectKeys in
  surveyDuplicate/categories tests; existing surveyVisibility suite untouched
  and passing.
- Attempt policy untouched: completed blocks restart (409), abandoned starts
  fresh; clone is intentionally an independent survey (documented).
- Migrations are add-only; 0001–0005 untouched.

3. Review pointers (highest-value places to look)
-----------------------------------------------------------------------

1. Soft-delete leak surface — the filter must hold at every participant
   path: surveyStructure.ts (includeDeleted default false),
   surveyAttempts.ts buildMySurveysResponse CTE (both branches),
   surveyAttemptRoutes.ts (start/answer/complete), surveyReadRoutes.ts
   (list always filtered; GET /:id includeDeleted only for admins).
   surveyDeletion.test.ts covers each; check for any query site I missed.
2. resolveAttemptPath refactor equivalence — compare the new loop bodies of
   determineCurrentQuestion / validateReachedRequiredQuestions against the
   pre-phase versions (git diff). Subtlety: the loop-error precedence in
   validateReachedRequiredQuestions (required-unanswered beats loop error,
   matching the old walk order) and determineCurrentQuestion returning null
   on loops.
3. Rule remap in duplicateSurveyTree — a missed remap silently corrupts
   skip logic. The test asserts all rule FKs resolve inside the clone; check
   the throw path and transaction rollback.
4. rejectDeletedSurvey middleware ordering — it is listed after
   requireAuth/requireRole on each builder route (never router-level, which
   would intercept the attempt router mounted after the builder router in
   surveys.ts). Verify no builder route lost the guard (POST "/" needs none).
5. SurveyAttemptPage extraction — the runner was moved from
   UserDashboard.tsx mostly verbatim; the new pieces are the mount-time
   resume logic, path-based Previous, stepper, and slider. The slider keeps
   "no selection" until interaction so required validation still bites.
6. Admin users SQL — confirm no password_hash in any projection.

Finding 1 (published surveys structurally mutable): FIXED.
- New rejectStructurallyLockedSurvey guard in surveyBuilderRoutes.ts on all
  14 structural routes (question add/update/delete/reorder, option
  add/update/delete/reorder, tag add/update/delete, rule add/update/delete).
  409 for any non-draft (published AND retired) survey:
  "Survey structure can only be edited while the survey is a draft. Create
  an editable draft copy to make changes". The guard also performs the

  option delete, plus the published type-change/scale-range checks in PUT
  question (hasScaleRangeChanged deleted from surveyBuilder.ts — verify no
  other consumer; grep is clean).

  survey is not a draft; SurveyLogicPage disables rule add/save/delete;
  published/retired banners now direct to "Create editable draft copy".
- Tests: new "published survey structural lock" describe block in
  surveyBuilder.test.ts sweeps all structural endpoints on a published
  survey (409 + unchanged structure), checks retired surveys, and confirms
  metadata/category/status changes still succeed. The old
