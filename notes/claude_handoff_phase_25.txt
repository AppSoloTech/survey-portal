## Phase Scope

Implemented:

- Added migration `0026_survey_timing_overrides.sql`.
  - `survey_timing_overrides.survey_id` is a one-to-one FK to `surveys`.
  - `admin_override_seconds` is positive and capped at 24 hours.
  - creator/updater user ids are stored for Admin audit context.
- Added shared contracts:
  - `SurveyTimingEstimateSource`
  - `SurveyTimingSummary`
  - `SurveyTimingResponse`
  - `Survey.effectiveEstimateSeconds`
- Added `apps/api/src/services/surveyTiming.ts`.
  - Backend default estimate is the sum of question-type weights with a 60-second minimum.
  - Weights: text 90s, integer 30s, single-select 45s, multi-select 60s, scale 30s.
  - Statistical estimate uses same-survey completed attempts only.
  - Valid samples require both timestamps, `completed_at > started_at`, and duration <= four hours.
  - Estimate uses the median valid duration.
  - Precedence is Admin override, then statistical, then default.
- Extended survey structure reads to attach only `effectiveEstimateSeconds` to every `Survey`.
- Added Admin-only timing APIs:
  - `GET /api/surveys/:id/timing`
  - `PUT /api/surveys/:id/timing`
  - `DELETE /api/surveys/:id/timing`
- Timing mutations are allowed on published surveys because this is metadata, not structural content.
- Added Admin Setup timing panel:
  - derived estimate
  - statistical sample count
  - default estimate
  - effective estimate
  - estimate source
  - override input in minutes
  - clear override action
- Added focused API tests covering derivation, invalid-sample ignoring, default fallback, override save/clear, precedence, authorization, published-survey override, and participant payload isolation.
- Updated `markdown/DATA_MODEL_VISION.md`, `markdown/FOLLOW_UPS.md`, and `markdown/PHASE_LOG.md`.

Non-goals respected:

- No participant progress-label replacement.
- No running/dynamic time remaining display.
- No per-page/per-question instrumentation.
- No per-question Admin overrides.
- No per-question-type Admin override settings.
- No analytics based on abandoned or in-progress attempts.
- No live dashboard or trend chart.
- No survey navigation, skip logic, reporting count, hidden tag, or CSV behavior changes.

Known tradeoffs:

- Statistical estimate uses a median and a four-hour cap. This is documented in the phase log as the first robust outlier policy while Phase 27 active-time instrumentation is pending.
- Default weights are code-owned constants for Phase 25; Admin-configurable per-question-type defaults are explicitly out of scope.
- `fetchSurveyTimingSummary` currently loads the survey structure and recomputes timing from the shared helper. This keeps behavior centralized, at the cost of one extra calculation in that endpoint.
- Participant-safe `Survey.effectiveEstimateSeconds` is now present on Admin and participant survey reads. Rich timing fields remain only on the Admin timing endpoint.

Deferred work:

- Manual browser pass was not run in this implementation turn.
- Phase 26 should consume `Survey.effectiveEstimateSeconds` and replace participant numeric progress labels.
- Phase 27 should add lightweight activity instrumentation and active-time aggregation.
