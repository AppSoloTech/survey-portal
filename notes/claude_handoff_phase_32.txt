## Phase Scope

Implemented:

- Added shared type `ConvertAnonymousSurveyAttemptResponse`.
- Moved backend registration body validation into
  `apps/api/src/services/validation.ts`.
- Kept normal `POST /api/auth/register` behavior and response shape unchanged.
- Added anonymous public endpoint:
  - `POST /api/anonymous-surveys/:token/register`
  - Requires normal registration fields, `attemptId`, and
    `attemptAccessToken`.
  - Reuses anonymous token/link availability validation through the existing
    anonymous owner resolver.
  - Uses a dedicated registration-grade rate limiter with the auth registration
    rate-limit window/max settings.
  - Locks the attempt row, requires completed anonymous ownership, creates the
    user, clears anonymous owner/contact fields, and sets the auth cookie.
  - Fetches the converted attempt with responses through the same transaction
    client before commit.
- Updated anonymous completion UI:
  - completed anonymous attempts now show an inline account-creation panel first
  - "Continue anonymously" opens the existing optional follow-up email modal
  - successful conversion updates auth context and redirects to `/dashboard`
- Added web API client function for conversion.
- Updated data-model docs, follow-ups, and phase log.
- Expanded `apps/api/test/anonymousSurvey.test.ts` from 5 to 7 tests.

Non-goals respected:

- No existing-user login or claim/merge flow.
- No password reset from the anonymous completion screen.
- No email verification or real email delivery.
- No anonymous invitation/recipient/list workflow.
- No public directory search/filter changes.
- No changes to anonymous start/answer/complete validation before completion.
- No conversion of in-progress, disabled-link, expired-link, retired-survey, or
  bad-token attempts.
- No hidden-tag, token hash, attempt token, password hash, or anonymous contact
  email exposure to participants.

Known tradeoffs:

- No schema migration was added. The existing `survey_attempts_owner_check` and
  anonymous contact-email check already allow conversion when anonymous fields
  are cleared.
- The conversion endpoint is under both the anonymous public route limiter and a
  dedicated registration-grade limiter.
- The conversion transaction revalidates and locks the attempt after the link is
  resolved. Link availability is rechecked before the transaction through the
  existing anonymous owner resolver.
- Browser/manual testing has not been run yet.

## Specific Questions

- Is the conversion transaction safe enough against partial user creation or
  partial ownership mutation?
- Should the endpoint additionally use a stricter auth-style registration rate
  limiter, or is the existing anonymous public route limiter sufficient?
- Are duplicate-email and invalid-registration failures guaranteed not to
  mutate the anonymous attempt?
- Does the conversion correctly clear anonymous owner/contact fields without
  losing responses or completion timestamps?
- Do dashboard, Admin Results, and CSV consistently classify converted attempts
  as registered-user attempts?
- Does the frontend flow make account creation the first post-completion choice
  while preserving the email fallback when declined?
- Is any password, token, hidden-tag, or anonymous contact email exposed in
  participant-visible responses or logs?
