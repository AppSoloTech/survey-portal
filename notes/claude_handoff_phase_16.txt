## Phase Scope

Implemented:

- Added placeholder-only email environment config with `EMAIL_ENABLED=false` and `EMAIL_PROVIDER=disabled` as the safe default.
- Added an internal provider-agnostic email client boundary in `apps/api/src/services/email.ts`.
- Added typed future template payloads for `password_reset` and `anonymous_survey_invite`.
- Added disabled and development no-op adapters only; neither sends real email.
- Added tests for disabled startup/config behavior, no-op send behavior, production guardrails, and sanitized no-op logging.
- Updated environment docs and follow-ups to keep real provider selection deferred.

Non-goals respected:

- No password reset routes, tokens, storage, or user-facing flows.
- No anonymous survey link generation, invitations, or public survey routes.
- No real email provider integration or credentials.
- No database changes.
- No queues, workers, or microservices.

Known tradeoffs:

- `EMAIL_ENABLED=true` is rejected in production until a real provider adapter exists, so production cannot accidentally run with a no-op provider that looks enabled.
- The no-op adapter logs only template metadata and recipient count. It does not log recipients, URLs, token strings, survey titles, or message bodies.
- The template payloads are minimal shapes for future callers; rendering and provider-specific transport are intentionally absent.

Deferred work:

- Choose an approved provider and credential strategy before real delivery.
- Implement password reset and anonymous survey invitation flows in their own scoped phases.
- Add provider-specific tests when a real adapter is approved.
