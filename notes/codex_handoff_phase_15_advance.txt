Source of the request
---------------------
Developer, after manually testing Skip page:
  "The skip page seems to work, but the action does not trigger until the rest of
   the questions on the page are answered. I would like the skip to occur once the
   trigger question and answer are submitted."
Clarified follow-up (developer's words): give the admin a toggle/lever to choose
  "skip remaining questions upon trigger" vs "only skip over the selected pages."

Decision settled before implementing
------------------------------------
- Make it an admin-controlled, per-rule boolean, NOT a global behavior change.
- Default OFF preserves the existing semantics (finish the trigger's page, then
  the targeted page is skipped at page transition).
- Lever applies to HIDE_PAGE only (it is meaningless for jumps and for
  HIDE_QUESTION); it is forced false for every other action type.

Behavior
--------
When a HIDE_PAGE rule has advance_on_trigger = true and its trigger answer is
submitted, the runtime stops revealing the rest of the trigger's (source) page
and advances to the next visible page, which skips the just-hidden target page(s).
This is the SAME immediate mid-page navigation that JUMP rules already perform
(progressivePageRunner "routes to a later page ... leaving later same-page
questions unrevealed"). The remaining same-page questions after the trigger become
off-path and are pruned by the existing pruneOffPathAnswers pipeline, exactly like
a jump-abandoned branch.

When advance_on_trigger = false (default), nothing changes: the trigger marks the
target page hidden, the source page is completed normally, and the target page is
skipped by getNextVisiblePage at the page transition.

Core design — where the runtime actually changes
------------------------------------------------
packages/shared/src/index.ts
  - ConditionalLogicRule gains `advanceOnTrigger: boolean`.
  - New exported helper hasActivatedAdvancingPageSkip(survey, question, response):
    true when answering `question` matches a HIDE_PAGE rule with advanceOnTrigger
    and a non-null targetPageId (reuses doesRuleMatchResponse, so equals + is_blank
    both work).
  - resolveProgressivePageState: after collecting the activated hidden question
    ids and AFTER the existing JUMP navigation check, a new branch:
        if (hasActivatedAdvancingPageSkip(survey, question, response)) {
          visibleQuestionIdsByPageId[page.id] = revealedQuestionIds;
          page = getNextVisiblePage(survey, page, activeHiddenQuestionIds);
          continue pageLoop;
        }
    Ordering matters: JUMP wins if both fire on the same question (jump is checked
    first and continues the loop). getNextVisiblePage already skips pages whose
    questions are all hidden, so the just-hidden target page is skipped.

KEY POINTS to scrutinize
------------------------
1. resolveProgressivePageState is the single resolver behind the live runner,
   the reporting keep-set (collectFinalPathQuestionIds), and pruneOffPathAnswers.
   So advancing also (correctly) prunes the now-off-path same-page questions after
   the trigger. Confirm that is the intended/acceptable data outcome (it mirrors
   jump-abandoned answers, which are already pruned).
2. The advance decision is OR-ed across a source question's HIDE_PAGE rules: if any
   matching advancing rule fires on the answered question, the page advances.
   Confirm that is sensible when a source has mixed advancing/non-advancing page
   skips (advancing wins).
3. resolveAttemptPath (the question-level walker) is intentionally NOT changed —
   advance-on-trigger is a page-flow concept and that walker does not model page
   navigation. Confirm nothing depends on it for HIDE_PAGE advance behavior.
4. Loop safety: advancing uses getNextVisiblePage + visitedPageIds, so the
   existing has-loop guard still applies.
