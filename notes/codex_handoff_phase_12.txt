Scope Implemented
-----------------
Frontend-only restructuring of the admin survey builder. No schema, migration,
API-contract, auth, or hidden-tag changes. All persistence uses pre-existing
endpoints.

1. Questions tab is now a focused, one-page-at-a-time editor.
   - New PageSwitcher (apps/web/src/components/admin/PageSwitcher.tsx): a pill
     rail reusing the .workspace-tab look + a "jump to page" <select>. Drives a
     local activePageId.
   - SurveyQuestionsPage.tsx: removed newQuestionPageId; added activePageId with
     a validity effect mirroring the prior one (keep current page if it still
     exists, else first page, else null). Renders only the active page's
     metadata edit form, the Add-question form (page <select> removed; page is
     implied by activePageId), and that page's QuestionEditor list (reused
     unchanged, keeps within-page Up/Down). Removed from this tab: the Add-page
     form, the per-question "Move to page" <select>, and the per-page
     Move up/down/Delete buttons (all now on Organize).

2. New "Organize" workspace tab (drag-and-drop board).
   - Route: apps/web/src/App.tsx (nested "organize" route after "questions").
   - Tab: SurveyWorkspaceLayout.tsx (WorkspaceTab "Organize (<pageCount>)").
   - SurveyOrganizePage.tsx: Add-page form + a @dnd-kit board. Sortable page
     cards (reorderSurveyPages on drop). Each card holds compact sortable
     question rows (OrganizeQuestionRow.tsx). Within-page drop -> reorderQuestions;
     cross-page drop -> moveQuestionToPage(displayOrder = dropIndex + 1, since the
     server treats displayOrder as 1-based). Page delete (existing guard: empty +
     >1 page) and per-row question delete are available. Keyboard drag via
     @dnd-kit KeyboardSensor + sortableKeyboardCoordinates; PointerSensor uses a
     6px activation distance so button clicks are not swallowed.

3. New dependency: @dnd-kit/core ^6.3.1, @dnd-kit/sortable ^10.0.0,
   @dnd-kit/utilities ^3.2.2 in apps/web (package.json + root lockfile).

4. apps/web/src/jsx-global-shim.d.ts: re-aliases the global `JSX` namespace to
   `React.JSX`. @types/react v19 removed the global JSX namespace; @dnd-kit's
   shipped type defs still reference it, so typecheck failed without this shim.

5. CSS appended to styles.css (section "10b"): .page-switcher(.-rail/-pill/
   -jump/.active/-count), .organize-board, .organize-page-card(.dragging),
   .organize-page-card-header/-heading/-questions, .compact-question-row
   (.dragging) + body/locator/text/type, .drag-handle, .organize-drag-overlay,
   plus mobile rules in the existing 767.98px block (pill rail horizontal scroll,
   compact-row wrap). Reuses existing design tokens.

Suggested Review Focus
----------------------
1. Drag-and-drop correctness in SurveyOrganizePage.handleDragEnd:
   - id namespacing (page:<id>, q:<id>, pagedrop:<id>) and resolveOverPageId().
   - same-page reorder uses arrayMove(ids, oldIndex, overIndex); confirm the
     resulting questionIds set is the full page list (server validates the set).
   - cross-page move passes displayOrder = targetIndex + 1 (1-based). Verify
     against the route at apps/api/.../surveyBuilderRoutes.ts (it splices at
     displayOrder-1 for same-page and inserts at displayOrder for cross-page).
   - dropping a question onto an empty page (the pagedrop droppable) and onto a
     page header (the page sortable) both resolve to that page, index = end.
2. Read-only gating: every structural control is disabled when
   survey.status !== "draft" (useSortable disabled, add/delete buttons disabled,
   handleDragEnd early-returns). Confirm a published/retired survey is fully
   viewable but immutable on both tabs.
3. activePageId validity effect in SurveyQuestionsPage: after deleting the active
   page on Organize or switching surveys, the Questions tab must fall back to a
   valid page (or the no-pages empty state).
4. The JSX global shim: confirm it is the minimal correct fix and does not mask
   real type errors (typecheck passes app-wide with it).
5. No participant-facing or data-model drift: Survey/SurveyPage/SurveyQuestion
   types are untouched; questions remain database-driven; hidden tags are still
   admin-only (the Organize board shows no tags).
