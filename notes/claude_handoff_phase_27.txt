## Phase Scope

Implemented:

- Added migration `0027_survey_attempt_activity_events.sql`.
  - Columns: attempt id, survey id, optional page/question context,
    `event_type`, `visible_question_ids`, `occurred_at`, and `created_at`.
  - Event type check allows only `page_entry`, `answer_save`, `resume`,
    `completion`, and `heartbeat`.
  - Indexes support attempt-time and survey-time reads.
- Added shared contracts:
  - `SurveyAttemptActivityEventType`
  - `SurveyAttemptActivityRequestPayload`
  - `SurveyAttemptActivityResponse`
- Added `apps/api/src/services/surveyActivity.ts`.
  - `surveyAttemptActivityIdleGapCapSeconds` is 300 seconds.
  - `recordSurveyAttemptActivity` inserts event rows.
  - `validateSurveyAttemptActivityContext` confirms page/question/visible
    question ids belong to the survey, with page scoping when a page id is sent.
  - `fetchAttemptActivitySummary` summarizes active seconds for one attempt.
  - `fetchSurveyActivitySummary` summarizes active seconds across all attempts
    for a survey.
- Added body validation for activity writes in `validation.ts`.
- Added authenticated endpoint:
  - `POST /api/surveys/:id/activity`
  - Requires `requireAuth`, validates attempt ownership through
    `fetchAttemptForUser`, rejects completed/abandoned attempts, validates
    context, records the event, and updates `last_activity_at`.
- Added anonymous endpoint:
  - `POST /api/anonymous-surveys/:token/activity`
  - Reuses anonymous public rate limiting and existing link + attempt access
    token ownership resolution, rejects completed/abandoned attempts, validates
    context, records the event, and updates `last_activity_at`.
- Added backend activity records for answer saves and completion in both
  authenticated and anonymous routes.
- Instrumented the shared participant runner:
  - resume event once per in-progress attempt load
  - page-entry event once per attempt/page
  - heartbeat every 60 seconds while visible and in progress
  - activity writes are fire-and-forget so they do not block participant flow
- Added `apps/api/test/surveyActivity.test.ts` covering:
  - authenticated attempt ownership validation
  - anonymous attempt token validation
  - completed/abandoned activity rejection
  - idle-gap capping and aggregation across page entry, answer save, resume,
    and completion events
  - participant payload isolation
- Updated:
  - `markdown/DATA_MODEL_VISION.md`
  - `markdown/FOLLOW_UPS.md`
  - `markdown/PHASE_LOG.md`
  - `notes/claude_handoff_phase_27.txt`

Non-goals respected:

- No complex prediction model.
- No Admin dashboards for activity timelines.
- No per-question Admin timing overrides.
- No CSV export of timing telemetry.
- No completion validation or navigation changes.
- No replacement of the Phase 26 remaining-time helper or copy thresholds.
- No raw text answers stored in timing events.
- No participant exposure of timing event internals.

Known tradeoffs:

- The idle gap cap is a conservative five minutes. It is centralized and
  documented for later tuning.
- Activity writes for answer saves and completion are recorded server-side
  instead of relying on the browser to emit those action events.
- Resume/page-entry de-duplication in the browser reduces noise, but it means
  repeated back-navigation to the same page is represented by heartbeat/resume
  context rather than unlimited page-entry rows.
- Completion event recording currently happens immediately after the completion
  update rather than as part of a refactored completion transaction. Review
  whether this is acceptable operational metadata behavior or should be
  transactional before merge.

Deferred work:

- Dynamic participant remaining-time prediction model remains a future phase.
- No Admin activity timeline or export was added.
- Manual browser pass remains pending.

  remaining-time display behavior unchanged?
