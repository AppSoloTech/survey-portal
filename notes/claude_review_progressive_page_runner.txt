1. CRITICAL ISSUES
================================================================================

None blocking. The prior same-page-jump completion defect is fixed and covered

================================================================================
2. SUGGESTED IMPROVEMENTS
================================================================================

S1 (high) Remove the now-dead, divergent resolveNextPage.
   The exported resolveNextPage (packages/shared/src/index.ts ~line 562) is no
   longer referenced by any code path — resolveAttemptPagePath now delegates to
   resolveProgressivePageState, which navigates via resolveTriggeredNavigationPage
   + getNextVisiblePage. resolveNextPage is a second, subtly different
   implementation of the same concept: it does not honor
   getNormalFlowExcludedPageIds (skipTargetInNormalFlow for JUMP_TO_PAGE) and is
   not jump-aware in the same way. Leaving it exported invites a future caller to
   pick up the wrong semantics. Delete it (and its dist/.d.ts surface) or collapse
   it into the progressive helpers so there is a single navigation source of truth.

S2 (medium) Completion error conflates "current question" with "required
   unanswered", and can be an optional question.
   validateReachedRequiredQuestions now returns
   `Required question ${state.currentQuestion.displayOrder} is unanswered`
   whenever state.currentQuestion is non-null. But currentQuestion is simply the
   first question lacking a *progressive* response, which for an optional question
   means "no row saved yet" (hasProgressiveResponseForQuestion returns false for
   an optional question with no response row). So:
   - The message can name an optional question as "required".
   - Semantically, completion is gated on every optional question having been
     explicitly Continue'd (row created), not just required ones.
   The UI prevents reaching Complete until currentPage is null, so this only
   surfaces on out-of-order API calls — but the message is misleading and the
   gating intent should be confirmed (see Q1). At minimum, only emit the
   "required" message when state.currentQuestion.isRequired.

S3 (medium) The page-batch endpoint is retained, unused, and untested.
   POST /api/surveys/:id/pages/:pageId/answer -> savePageAnswers is kept "for
   compatibility", but the web runner now saves through the single-question
   /answer endpoint, and no test exercises the batch path. savePageAnswers also
   writes a row for *every* question on the page, so it can persist off-path
   answers when an earlier answer on the page branches away (progressive reveal
   would never have shown those questions). Either add coverage and a comment
   marking it compatibility-only, or drop it (removal is a stated non-goal, so a
   test + comment is the safer choice).

S4 (medium) Add HTTP-layer tests for the genuinely new behavior.
   Shared unit coverage is good, but the API integration tests still model only
   one-question-per-page surveys (createPublishedJumpSurvey / createPublishedSkip
   Survey were migrated to one page per question, which keeps attemptLifecycle
   passing). There is no API test for: a multi-question page revealing
   progressively through /answer, the currentPageQuestionIds contract field,
   editing an earlier answer to flip a branch and dropping later same-page
   answers, or the batch endpoint. factories.addPage + the pageId option make
   these straightforward to add now.

S5 (low) Previous-page navigation has no forward affordance.
   onPrevious sets currentQuestion: null, so the prior page renders every question
   with an "Update" button and none as the active "Continue" step. The only way
   back forward is to press Update on an already-answered question, which
   re-saves it and (via recompute) jumps the participant all the way to their
   furthest unanswered point rather than to the next page. Consider an explicit
   Next/Resume control, or restoring the active step on the returned page.

S6 (low) Repeated re-sorting. getOrderedQuestions/getOrderedPages re-sort on every
   getQuestionsForPage call, which runs inside the per-page/per-question loops of
   resolveProgressivePageState and again per answered question in
   resolveTriggeredNavigationPage; the web page also calls resolveAttemptPagePath
   three times per render. Negligible at MVP sizes; memoize the ordered lists if
   surveys grow.

S7 (low) "Farthest later page wins" across multiple questions is largely
   unreachable. Because reveal is single-step, the first revealed question that
   triggers a jump exits the page before later questions are revealed, so
   resolveTriggeredNavigationPage almost always reconciles rules from a single
   question. The cross-question candidate gathering is effectively dead except for
   multiple matching rules on one question. Worth a comment, or simplify.

================================================================================
3. ACCEPTABLE TRADEOFFS
================================================================================

- Legacy jump/skip fixtures rewritten as one-question-per-page surveys — a clean
  way to keep existing attemptLifecycle assertions valid under the page model and
  representative of migrated legacy surveys.
- JUMP_TO_QUESTION resolves only to its containing page (no intra-page focus);
  same-page jumps are no-ops. Matches the stated non-goals.
- Stale off-path answers are kept in the DB and excluded from the active path
  (resolveProgressivePageState walks only reachable pages). Consistent with the
  existing resolveAttemptPath approach.
- "Pages on your path" is a normal-order projection (appendProjectedPagePath) and
  may shift as jumps fire on later answers. Acceptable estimate while in progress;
  at completion the path contains only actually-walked pages (the terminal,
  all-answered return does not project).
- Optional questions require an explicit Continue (row creation) to advance.
  Reasonable for a one-at-a-time runner; see S2/Q1 for the completion-gating angle.
- Progressive save reuses the single-question /answer endpoint; no new migration.

================================================================================
4. QUESTIONS / ASSUMPTIONS
================================================================================

Q1. Should an un-answered *optional* question block Complete? Today it does
    (currentQuestion is set until a row exists), and the UI enforces clicking
    through every optional question. Confirm this is intended; if not, treat
    optional questions as satisfied for completion (S2).

Q2. Is the Previous-page interaction intended to require re-submitting an answer
    to move forward, and to jump to the furthest point rather than the next page
    (S5)?

Q3. Keep the unused page-batch endpoint as compatibility surface, or retire it?
    (S3 — removal is currently a non-goal.)

Q4. First-page edge: if a survey's first page consists entirely of
    normal-flow-excluded questions (skipTargetInNormalFlow JUMP_TO_QUESTION
    targets), the main loop pushes that empty first page onto the path with no
    revealed questions. Can publish validation produce such a survey? If so, it
    inflates the page count and shows an empty first screen until a jump fills it.

Q5. Hidden-tag boundary unchanged — participant fetches still use
    includeHiddenTags: false (buildMySurveyResponse and both answer routes), so
    currentPageQuestionIds / currentPage carry no tag data. Assumption confirmed
    in code; flagging only so it stays a checked invariant.
