Scope Implemented
-----------------

1. Service extraction from apps/api/src/routes/surveys.ts (was 3,650 lines;
   now a 19-line router composition). Zero intended behavior change.
2. First automated test harness: Vitest in all workspaces, supertest API
   route tests against a dedicated PostgreSQL test database. 93 tests total.
3. Admin reporting + CSV export: four new admin-only endpoints, shared
   types, a Results tab in the survey workspace, and an overview completion
   indicator.

Product decisions (human said: apply recommended defaults, document):
- One attempt per user per survey stands; the attempt is the reporting unit.
- Branch-orphaned answers are kept and marked "not on final path"
  (on_final_path column in CSV; badge in attempt detail). Path computed by
  walking shared resolveNextQuestion over saved responses.
- Blank optional rows reported as "skipped (blank)"; absent rows are
  "not answered yet" (on path) or "never reached" (off path).

Suggested Review Focus
----------------------

1. Hidden-tag and PII boundaries: reporting responses and the CSV expose
   hidden tags and participant emails by design — verify every reporting
   route enforces requireAuth + requireRole("admin") and that no participant
   path gained tag access. Tests assert this (reporting.test.ts
   "reporting authorization", surveyVisibility.test.ts), but a second pair
   of eyes on the route wiring is the point of review.
2. The final-path walk (collectFinalPathQuestionIds in surveyReporting.ts):
   confirm it faithfully mirrors resolveNextQuestion semantics, including
   the loop guard for defensive legacy data.
3. The extraction's mount order in routes/surveys.ts: read → builder →
   attempts → reporting. Route patterns don't overlap across methods/paths,
   but please double-check nothing shadows GET /:id vs GET /:id/report.
4. Test DB safety: test/helpers/testDatabaseUrl.ts must make it impossible
   to point tests at a real database. Note the documented caveat that a
   bare `vitest` at the repo root races all projects against one DB —
   acceptable, or worth hardening now?
5. CSV correctness: services/csv.ts quoting (RFC 4180) and the export's
   one-row-per-saved-response shape (not_reached questions are omitted).
6. The overview completion indicator fetches one report per non-draft
   survey (N+1). Accepted as MVP-scale and tracked in FOLLOW_UPS — flag if
   you disagree.
