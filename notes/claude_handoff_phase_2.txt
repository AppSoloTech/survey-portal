  - `GET /api/surveys`
  - `GET /api/surveys/:id`

- Updated the Claude handoff template so future phase implementations explicitly require a handoff before completion.

Non-goals:

- Survey runner UI.
- Admin survey builder UI.
- Runtime conditional logic evaluation.
- Analytics.
- Reporting.
- CSV export.
- Question/answer option/rule editing endpoints beyond foundational survey create/update.
- Implementation of `SHOW_QUESTION`, `HIDE_QUESTION`, `JUMP_TO_PAGE`, or `END_SURVEY` behavior.

Known tradeoffs:

- `POST /api/surveys` and `PUT /api/surveys/:id` currently create/update survey metadata only. The normalized child tables are read and seeded, but question/option/rule management endpoints are left for later admin-builder phases.
- Public `GET /api/surveys` and `GET /api/surveys/:id` expose conditional rule records for published surveys, but do not expose hidden answer tags. This supports future survey flow setup while preserving hidden tag privacy.
- Admin callers can see all survey statuses and hidden tags when they provide a valid admin bearer token.
- The SQL migration remains ordered/raw SQL because the migration runner follow-up is still active.
- Seed data is local fake data only and includes a documented local password.

Deferred work:

- Choose a database migration runner before deployment workflow becomes repetitive.
- Add route-level tests for survey APIs and auth visibility rules.
- Add dedicated admin endpoints for question, option, tag, and rule management in a later admin-builder phase.
- Revisit local-storage JWT versus httpOnly cookie auth before real survey response data or hidden-tag metadata exists.
- Add auth route rate limiting before public exposure.
- Update `/api/health` readiness semantics before Azure health checks rely on it.
