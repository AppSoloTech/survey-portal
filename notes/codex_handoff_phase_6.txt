## Summary

The single-page admin builder is split into a surveys overview at `/admin` and
a per-survey workspace at `/admin/surveys/:surveyId` with four routed pages:
setup, questions, logic, and preview. The selected survey moved from component
state into the URL. This is a frontend-only restructure: no schema, API, or
auth changes; one thin frontend fetch helper was added for the existing
`GET /api/surveys/:id` endpoint. The old `AdminDashboard.tsx` is deleted and
its handlers were moved (not rewritten) into the new pages.

## Route And State Design

- Routes nest under the existing `AdminRoute` guard; server-side admin
  authorization is unchanged.
- `SurveyWorkspaceLayout` validates `:surveyId` (positive integer), fetches
  the survey, and renders loading / not-found states. Children receive
  `{ survey, isSubmitting, runSurveyMutation, changeStatus, setFeedback,
  customTagPresets, setCustomTagPresets }` via outlet context.
- `runSurveyMutation` is the same pattern as before: clear feedback, run the
  API call, replace the survey object from the response, set the scoped
  success message. Because all pages derive from that one object, the flow
  map and tab counts update automatically after any mutation.
- Status actions (publish/retire/republish, retire confirm, identical
  feedback strings) live in the layout header on every page; the setup page
  additionally shows the availability detail panel.

## Intentional Behavior Notes (please review)

1. Cross-survey "Saved in surveys" tag suggestions previously came from the
   sidebar's full survey list. The questions page now fetches
   `fetchAdminSurveys` once per mount for suggestion derivation, merging the
   live workspace survey over its stale snapshot; on fetch failure it silently
   falls back to current survey + defaults + custom. One extra GET per
   questions-page visit.
2. Feedback messages persist across tab switches until the next action
   (matches the old single-page behavior where messages stayed at the top).
3. The bottom "Finish construction" status panel and back-to-top link were
   removed rather than relocated: pages are short now and header status
   actions are persistent. The availability panel remains on setup.
4. The old auto-select-first-survey behavior is gone by design; `/admin` is a
   list and the workspace is URL-addressed.
5. Anchor ids (`survey-preview`, `survey-logic` etc. removed from nav) — the
   remaining ids on sections are inert and kept only as harmless markup.

## Post-Review Fix Addendum

Codex review was recorded in `notes/codex_review_phase_6.txt` with one Medium
finding: `runSurveyMutation` lacked the stale-response guard that the fetch
effect has, so a mutation started on one survey could apply its result
(`setSurvey`, feedback, submit state) after navigating to a different survey's
workspace.

Fix applied in `SurveyWorkspaceLayout.tsx`:

- Added `activeSurveyIdRef`, updated at the top of the surveyId effect
  (including the null/invalid-id path).
- `runSurveyMutation` captures the survey id at call time and only applies
  `setSurvey`/`setNotice`/`setError` — and only clears `isSubmitting` — while
  the layout still shows that survey; stale results are dropped and return
  false.
- Switching surveys also resets `isSubmitting` in the fetch effect so an
  abandoned in-flight mutation cannot leave the new workspace's controls
  disabled.

Overlapping mutations on the same survey keep their previous last-response-wins
behavior, unchanged from the single-page builder.

Re-validated after the fix: `npm run typecheck`, `npm run lint`,
`npm run build`, and `git diff --check` all passed from the repo root.
