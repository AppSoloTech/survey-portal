## Phase Scope

Implemented:

- Top-level `loadtest/` harness.
- `.env.loadtest.example` with placeholders only.
- Root npm scripts:
  - `loadtest:seed`
  - `loadtest:run`
  - `loadtest:db`
  - `loadtest:teardown`
  - `loadtest:doctor`
  - `test:loadtest`
- `.gitignore` updates for load-test env, manifests, and reports.
- Explicit `.env.loadtest` parsing and host safety guards:
  - localhost targets require `--dev`
  - hosted writes require confirmation or `--yes`
  - hosted DB uses TLS config with `rejectUnauthorized: true`
- Seed script that creates namespaced fake Admin/user/survey/question/attempt/link data and writes an ignored manifest.
- Teardown script that defaults to dry-run unless `--yes` is passed and deletes only manifest/marker-matched data.
- k6 scenarios for read-heavy, write-heavy anonymous, mixed, and smoke profiles.
- Result persistence to `performance_test_runs`, including `running` start and completed/failed finish updates.
- Direct DB optional load check.
- PostgreSQL metrics sampling from `pg_stat_activity` and `pg_stat_database`.
- Summary normalization, conservative bottleneck classification, and markdown report generation.
- Operator README, release note, follow-up, and phase log updates.

Non-goals:

- No Admin UI pages.
- No browser controls to run, stop, or schedule tests.
- No App-Service-hosted load generation.
- No Azure Container Apps, Container Instances, Azure Load Testing, queues, functions, Blob Storage, or new cloud dependencies.
- No required Azure metric sampling.
- No live-network automated tests.
- No huge raw k6 event streams in PostgreSQL.
- No production app runtime behavior changes.

Known tradeoffs:

- Azure Monitor metrics are labeled unavailable; no Azure CLI calls are made in Phase 50.
- `--persistence-smoke` exists for safe persistence validation without k6 or HTTP load.
- The k6 anonymous write flow dynamically answers current pages and completes fresh anonymous attempts; it intentionally avoids registered-user repeat completion.
- After live k6 validation, page-submit scenarios answer every required question on the submitted page and include the UI-compatible `isOtherSelected`/`otherText` fields.
- The direct DB load query is report-like but intentionally simpler than the full Admin report service. It is meant to distinguish broad DB pressure, not duplicate app code.
- `.env.loadtest` is ignored and was created locally with fake localhost/test settings for smoke validation; it should not appear in `git status --short`.

Deferred work:

- Optional Azure Monitor sampling if operators later need App Service CPU/memory/resource metrics.
- Phase 51 read-only Admin performance report viewer.
- Time-series sample storage remains deferred until UI/report needs prove it is necessary.

## Specific Questions

1. Do target guards prevent accidental localhost/hosted confusion?
2. Is `.env.loadtest` parsing appropriately isolated from app `.env` and shell env?
3. Is teardown narrow enough to avoid deleting non-load-test data?
4. Does the k6 write-heavy scenario correctly use fresh anonymous attempts and avoid registered-user lifecycle conflicts?
5. Do result persistence updates correctly set `updated_at = now()` when completing/failing rows?
6. Does SQL metric sampling degrade safely when permissions are missing?
7. Is the `--persistence-smoke` path acceptable as an explicit non-load validation mode?
8. Did this introduce any Azure orchestration or in-app load-runner scope by accident?
