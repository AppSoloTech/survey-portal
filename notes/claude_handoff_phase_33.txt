## Non-Goals

- No in-app editing for release notes.
- No database storage or migrations.
- No public release-note page.
- No git commit/changelog generator.
- No broad admin navigation redesign.

## Known Tradeoffs

- Release-note parsing is intentionally strict and supports a small fixed
  section set: Added, Changed, Fixed, Security, Operational Notes.
- `release:check --since <ref>` compares committed `HEAD` against the ref.
  It is expected to fail before the phase is committed because uncommitted files
  are not part of `HEAD`; `npm run deploy` already requires a clean tree before
  invoking this mode.
- Root `package.json` is the app release version source of truth. Workspace
  package versions remain `0.1.0` because their package contracts did not
  change.
- Draft notes in `markdown/releases/unreleased.md` are not shown in the admin
  UI because the API only reads versioned `vX.Y.Z.md` files.
