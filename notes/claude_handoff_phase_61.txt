# Claude Handoff — Phase 61 Category All Review Tags

## Summary

Phase 61 adds an admin-only virtual category `<ALL>` action for manual review
tags on text answers in Survey Results, and applies the same living
subscription model to builder hidden tags in the Questions tab.

The implementation intentionally does not create real `<ALL>` rows in
`tag_definitions`. The Results dropdown synthesizes `<ALL> - Apply all in {category}`
options from the current tag groups, and the API stores a category-level
binding plus expands the selected category into real tag definition IDs.

Product clarification after initial review: subscribed answers should maintain
category parity. Future tags added or moved into a category must propagate to
answers where that category `<ALL>` selector was already applied, and tags moved
out of or removed with the category should be removed unless they were manually
added. The same product expectation now applies to builder hidden tags: option,
Other, and value-condition hidden-tag targets that use `<ALL>` should stay in
parity with future Admin Tags catalog changes.

## Semantics

- `<ALL>` is virtual UI/API behavior, not a catalog row.
- On the Questions tab, selecting hidden-tag value `<ALL>` stores one
  `hidden_tag_all_bindings` row for the target and inherited source rows in
  `hidden_tag_all_binding_tags`, then expands all current real catalog values
  into the effective hidden-tag table for that target.
- Builder hidden-tag `<ALL>` is a living category binding keyed by `tag_key`.
  Future Admin Tags values with the same `tag_key` propagate to all subscribed
  option, Other, and value-condition targets, including published surveys and
  already submitted attempts because reporting reads the current hidden-tag
  metadata.
- Deleting or editing a catalog value removes inherited-only hidden tags from
  subscribed builder targets. Manual hidden tags are preserved.
- Stopping a builder hidden-tag `<ALL>` subscription deletes the binding and
  inherited source rows, then removes inherited-only effective hidden tags for
  that target.
- Migration `0042` backfills old marker targets with current catalog values,
  and migration `0043` converts those old marker targets into real
  subscriptions.
- Selecting `<ALL> - Apply all in {category}` stores one
  `response_answer_tag_groups` binding, inherited source rows in
  `response_answer_tag_group_tags`, and effective real tag chips in
  `response_answer_tags`.
- The action is a living category binding.
- Future tags created, edited, or moved into a bound category are inserted onto
  all bound answers, including submitted/completed attempts.
- Tags moved out of a bound category are removed from bound answers when they
  were inherited-only.
- Deleting a bound category removes inherited-only expanded review tags before
  the binding and inherited source rows cascade away.
- Manual review tags are sticky. If a tag is both manual and inherited, moving
  it out of the category, deleting the category, or stopping auto-apply removes
  only the inherited source and leaves the visible manual tag.
- Existing `response_answer_tags(answer_id, tag_definition_id)` uniqueness makes
  visible chips idempotent, `response_answer_tag_groups(answer_id, group_id)`
  makes binding idempotent, and
  `response_answer_tag_group_tags(answer_id, group_id, tag_definition_id)` makes
  inherited sources idempotent.
- If a category binding already exists for an answer, the category all option
  is hidden.
- Stopping category auto-apply deletes the binding and inherited sources, then
  removes inherited-only visible tags for that category.
- Empty groups can be bound so future category tags propagate.
- Ungrouped tags keep their existing individual options and have no virtual all
  selector.

## Review Focus

- Confirm no real `<ALL>` tag definitions are created or needed.
- Confirm `response_answer_tag_groups` and `response_answer_tag_group_tags` have
  the right foreign keys, cascade behavior, and uniqueness.
- Confirm the new category endpoint keeps the same admin-only authorization and
  answered-text validation as individual review tags.
- Confirm tag create/edit/move propagation occurs inside the catalog mutation
  transaction, syncs move-in and move-out behavior, and remains idempotent.
- Confirm category deletion removes inherited-only expanded tags while
  preserving manual tags.
- Confirm stopping auto-apply removes inherited-only tags but preserves manual
  tags.
- Confirm wrong survey/attempt/answer ownership remains protected.
- Confirm `<ALL>` never appears as a review chip, CSV value, report aggregate,
  tag catalog row, or Admin Tags management item.
- Confirm the dropdown label and sentinel value are clear and do not collide
  with numeric tag IDs.
- Confirm living binding behavior is documented and tested, including submitted
  answers.
- Confirm builder hidden-tag add forms expose visible `<ALL>` values for option,
  Other, and text/integer value tags without persisting `<ALL>` as a hidden tag.
- Confirm builder hidden-tag `<ALL>` creates `hidden_tag_all_bindings` rows and
  managed effective hidden-tag rows with `isManual: false`.
- Confirm Admin Tags create/edit/delete operations propagate to builder
  hidden-tag subscriptions, including submitted attempts, while preserving
  `isManual: true` hidden tags.
- Confirm older marker rows are converted by migration `0043` and no `All` or
  `<ALL>` hidden-tag values remain after migration.
