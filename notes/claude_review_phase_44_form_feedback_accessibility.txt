Findings:

1. Medium - Toast live-region reliability

Initial issue:
Success toasts were rendered as newly inserted `role="status"` elements inside
a container without persistent live-region semantics. Some assistive
technologies may not announce polite status nodes that appear with their text
already populated.

Resolution:
Codex updated `ToastProvider` to keep persistent live-region containers: one
polite `role="status"` region for success toasts and one assertive
`role="alert"` region for error toasts. Toast rows remain normal content with a
dedicated dismiss button.

2. Low - Disabled button descriptions were inert

Initial issue:
`aria-describedby` was attached to natively disabled reset controls. Native
disabled buttons are not focusable, so the description association would not be
announced from the control.

Resolution:
Codex changed the invalid reset-link and account reset-cooldown controls to use
focusable `aria-disabled` states where the explanation matters, kept native
`disabled` for submitting states, and guarded the handlers.

3. Low - Migrated auth fields lost staggered reveal animation

Initial issue:
The old auth labels had `data-reveal`; migrated `FormField` labels did not,
causing inputs to appear before the staggered button/note animation.

Resolution:
Codex added a `reveal` prop to `FormField` and enabled it for migrated auth
fields.

4. Nit - Required marker text is part of the accessible name

Status:
Accepted as intentional for now. It provides visible and programmatic
requirement context. Optional markers also remain in the label, which is useful
because native HTML has no `optional` equivalent.
