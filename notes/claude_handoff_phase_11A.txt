## Implemented Scope

- Added `database/migrations/0013_survey_pages.sql`.
- Added `SurveyPage`, `Survey.pages`, `SurveyQuestion.pageId`, and `currentPage` response fields.
- Added page-aware shared helpers: `getOrderedPages`, `getOrderedQuestions`, `getQuestionsForPage`, `resolveNextPage`, and `resolveAttemptPagePath`.
- Updated survey structure loading, record mapping, duplication, publish validation, rule validation, and attempt completion validation.
- Added admin page CRUD/reorder, page-scoped question reorder, and question move endpoints.
- Added participant page batch answer endpoint: `POST /api/surveys/:id/pages/:pageId/answer`.
- Updated admin Questions UI to manage page panels.
- Updated admin Logic UI and flow map for `JUMP_TO_PAGE`.
- Updated participant runner to show and submit one page at a time.
- Kept legacy `Survey.questions`, `currentQuestion`, and `/api/surveys/:id/answer` for compatibility.

## Non-Goals

- No sections separate from pages.
- No backward or same-page page jumps.
- No `SHOW_QUESTION`, `END_SURVEY`, or page-level hide actions.
- No removal of the legacy single-question answer endpoint.
- No public anonymous survey links.

## Known Tradeoffs / Review Hot Spots

- The flow map still visualizes question nodes; page jumps point to the first question on the target page.
- `Survey.questions` remains flattened for compatibility, sorted by page order then question order.
- Participant page batch submit saves blank rows for unanswered optional questions on the visible page, matching the existing optional-question flow.
- API tests were not run because the harness drops/recreates the local PostgreSQL test schema and sandbox escalation was rejected pending explicit user approval.
