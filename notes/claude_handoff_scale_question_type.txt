## Data Model Notes

SCALE intentionally reuses existing answer-option/tag infrastructure:

- `survey_questions.question_type = 'scale'`
- generated numeric values live in `answer_options`
- hidden tags live in `answer_tags`
- selected value tags resolve through `survey_response_selected_options`
- selected numeric value is mirrored into `answer_integer`

Please evaluate whether mirroring the numeric value into `answer_integer` is an acceptable reporting convenience or whether it risks redundant state issues.

## Specific Review Questions

1. Does using generated `answer_options` for numeric scale values preserve the Hidden Tag Principle?
2. Does the implementation avoid exposing hidden tags to participants?
3. Are the scale range validation rules sufficient and located on the server?
4. Is the draft-only range-change restriction enough to protect published response history?
5. Are scale responses stored in a way that supports future reporting and export?
6. Does deriving `scaleMin` and `scaleMax` from generated options create any API consistency risk?
7. Are conditional rules intentionally limited to `single_select` and `multi_select`, or should `scale` also be eligible?
8. Does the admin UI make clear that scale values are generated and only tags are edited per value?
9. Is the customer scale rendering usable across narrow and desktop widths?
10. Does migration `0005` safely update the existing check constraint?

## Known Tradeoffs

- Scale min/max are not stored as dedicated columns. They are derived from generated numeric `answer_options`.
- Scale values are not eligible as conditional-rule sources in this implementation because existing conditional logic is restricted to `single_select` and `multi_select`.
- Changing a draft scale range may delete generated option rows outside the new range, which also deletes their tags. This is expected for draft surveys but should be reviewed for admin UX clarity.
- There are no new dedicated automated tests because this repo currently relies on typecheck, lint, build, and manual validation patterns.
