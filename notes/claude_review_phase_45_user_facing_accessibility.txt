2. SUGGESTED IMPROVEMENTS

(a) [Medium] Category empty state now "breathes" like a loading indicator.
    CategorySurveysPage renders the terminal empty state as
    <AlertMessage className="builder-empty-state" variant="info">. For
    variant="info", AlertMessage emits class "status muted builder-empty-state".
    The rule .status.muted (styles.css:949) applies
    `animation: status-breathe 1.8s ease-in-out infinite`, which exists
    specifically for loading lines ("Loading lines breathe instead of sitting
    flat"). The category empty state is NOT a loading state - it is a terminal
    "no surveys in this group" message - so it will now pulse indefinitely,
    which is semantically misleading and visually distracting.
    Note: a global prefers-reduced-motion block (styles.css:5279) neutralizes
    the animation for reduced-motion users, so this is a UX/polish issue, not a
    hard a11y violation. Recommend a non-animated info treatment for terminal
    states (e.g. an AlertMessage variant or class that does not inherit
    .status.muted), or keep the empty state as its own primitive and only adopt
    the polite live-region role.

(b) [Low] Conflicting display/border declarations resolved only by source order.
    Stacking "status muted builder-empty-state" on one <p> means
    .builder-empty-state (styles.css:2940, display:grid, dashed border,
    radius-md, 1rem padding, surface bg) silently overrides the .status
    equivalents (line 936, display:inline-flex, solid border, radius-sm) purely
    because it is declared later at equal specificity. The current visual result
    happens to match the old block layout, but it is fragile - reordering or
    re-theming .status would change the empty state unexpectedly. A dedicated
    class would be more maintainable than relying on cascade order.

(c) [Low / optional] The {" "} text node between <strong> and <span> in the
    converted empty state is inert under display:grid (whitespace-only nodes are
    ignored as grid items). Harmless, but it signals the markup was shaped for an
    inline-flow layout that the grid display does not actually use. Drop it or
    confirm the intended layout.

(d) [Nit] Home.tsx "View anonymous surveys" is a single, non-repeated link, so
    the visually hidden " in the public directory" suffix is not strictly needed
    for disambiguation (unlike the repeated card/list actions, where it is
    valuable). It is still a reasonable clarity improvement; flagging only so the
    pattern is applied intentionally rather than uniformly.


3. ACCEPTABLE TRADEOFFS

(a) Source-level (readFileSync + toContain) tests. They do not render DOM or
    simulate AT, but they match the established project convention
    (AccessibleModal.test.ts, FormFeedback.test.ts) and the toolchain has no
    jsdom/RTL. Consistent and acceptable; just be aware they assert source
    strings, not rendered accessibility behavior, and break on benign
    refactors.

(b) Glossary aria-describedby now permanently targets the popover, with the
    popover kept mounted and toggled via hidden={!isOpen}. This is the correct
    and idiomatic improvement: aria-description (ARIA 1.3) had effectively no
    AT support, whereas aria-describedby -> a role="tooltip" element that is
    removed from the accessibility tree while hidden is well-supported. When
    closed, the HTML hidden attribute keeps the description out of the a11y tree
    (no premature announcement); when open it is exposed. Keyboard (Escape +
    blur), pointer (hover enter/leave), touch, focus, and click-toggle behavior
    are unchanged - only the render condition changed. Good call.

(c) Visually hidden contextual suffixes. Accessible names become e.g.
    "Resume survey: <title>", "Start survey: <title>", "View surveys in
    <category>" while the visible labels stay short. This is the standard
    technique for disambiguating repeated actions and does not alter the visible
    UI. Verified .visually-hidden (styles.css:5262) is a correct clip/inset
    off-screen utility, not display:none, so the text stays in the a11y tree.


4. QUESTIONS OR ASSUMPTIONS

(a) Is the breathing animation on the category empty state (2a) intended? I am
    assuming not - it reads as an oversight from reusing variant="info"
    (loading) styling for a terminal state.
(b) Assumption: the category empty-state copy change ("No surveys in this
    group." with trailing period, and "Back to dashboard.") is an intentional
    microcopy adjustment, not accidental.
(c) The glossary trigger retains aria-expanded (pre-existing, unchanged). For a
    pure tooltip pattern aria-expanded is non-standard, but since it predates
    this phase I treat it as out of scope.
