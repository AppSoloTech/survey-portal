## Summary

Phase 62 fixes the Admin Tags create workflow so a new tag inherits the existing
tag-key category emoji when the Admin leaves the emoji field blank and exactly
one distinct non-null emoji exists across matching `tag_definitions.tag_key`
entries. Matching rows with null emoji do not block inheritance.

The implementation is intentionally server-side. The Admin Tags UI already
sends `null` for a blank emoji field, so the backend now treats null/omitted
create-time emoji as "inherit a safe category default if one exists."

## Implementation Details

- Added `fetchCommonTagKeyEmoji(queryable, tagKey)` in
  `apps/api/src/routes/tags.ts`.
- The helper queries distinct non-null emoji values for the matching
  `tag_key`.
- `POST /api/tags` now uses:
  - the explicit request emoji when non-null
  - otherwise the common tag-key emoji when exactly one exists
  - otherwise `null`
- No schema changes, migrations, new endpoints, or UI controls were added.
- `PUT /api/tags/category-emoji` remains unchanged and continues applying or
  clearing emoji on existing definitions for a `tagKey`.
