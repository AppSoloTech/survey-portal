## Phase Scope

Implemented:

- Added shared timing weights for participant remaining-time proportions:
  - text 90s
  - integer 30s
  - single-select 45s
  - multi-select 60s
  - scale 30s
- Updated `apps/api/src/services/surveyTiming.ts` so backend default survey
  estimates use the same shared weight lookup. This avoids backend/frontend
  timing-weight drift.
- Added `calculateSurveyRemainingTimeEstimate`.
  - Uses `Survey.effectiveEstimateSeconds` as the total estimate.
  - Uses `resolveAttemptPagePath` output, including `visibleQuestionIdsByPageId`.
  - Scales remaining visible question weight against the visible path's total
    question weight.
  - Counts unanswered visible current-page questions as remaining.
  - Counts all visible future-page questions as remaining.
  - Uses the existing progressive response semantics for whether a current-page
    question has been answered.
- Added `formatRemainingTimeCopy`.
  - `<= 15s`: `Almost done`
  - `< 60s`: `Less than 1 min remaining`
  - otherwise: rounded `About N min remaining`
- Updated the participant runner.
  - Replaced visible `Page X of Y` heading with remaining-time copy.
  - Kept the visual progress bar, but made it decorative with `aria-hidden`.
  - Added `aria-live="polite"` to the remaining-time heading.
  - Applies equally to authenticated and anonymous runner modes because they
    share `SurveyRunner`.
- Added focused tests:
  - `packages/shared/test/surveyRemainingTime.test.ts`
  - `apps/web/src/pages/SurveyAttemptPage.test.ts`
- Updated:
  - `markdown/FOLLOW_UPS.md`
  - `markdown/PHASE_LOG.md`
  - `notes/claude_handoff_phase_26.txt`

Non-goals respected:

- No Admin timing override controls.
- No statistical timing derivation changes.
- No per-attempt activity instrumentation.
- No dynamic elapsed-time estimate.
- No survey navigation changes.
- No dashboard/Admin pagination label changes.
- No completion-screen total elapsed-time display.

Known tradeoffs:

- The estimate remains approximate. It distributes the survey-level effective
  estimate across question-type weights and the currently resolved visible path.
- Future branching behind unanswered questions is necessarily projected by the
  existing path helpers; Phase 27 is still the place for running/dynamic
  activity instrumentation.
- Manual browser checks were not run in this implementation turn.

Deferred work:

- Manual Phase 26 browser pass remains pending:

  - branch/page-jump/skip updates estimate after path changes
  - multi-question page accounts for remaining visible questions
  - no visible participant `Page X of Y` or `Question X of Y`
  - Admin pagination labels still render normally
  - anonymous public survey matches authenticated behavior
  - responsive checks at 375/768/1280px
