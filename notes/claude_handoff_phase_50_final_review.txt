## Phase Scope

Implemented:

- Top-level `loadtest/` operator harness.
- `.env.loadtest.example` with placeholders only.
- Root npm scripts:
  - `loadtest:seed`
  - `loadtest:run`
  - `loadtest:db`
  - `loadtest:teardown`
  - `loadtest:doctor`
  - `test:loadtest`
- Explicit `.env.loadtest` parsing and host safety guards:
  - localhost targets require `--dev`
  - hosted writes require confirmation or `--yes`
  - hosted DB uses TLS config with `rejectUnauthorized: true`
- Seed script for fake namespaced Admin/user/survey/question/attempt/link data.
- Teardown script with dry-run default and marker/manifest-scoped deletion.
- k6 scenarios for smoke, read-heavy, write-heavy anonymous, and mixed profiles.
- Ramping VU stages for non-smoke profiles.
- Result persistence to Phase 49 `performance_test_runs`.
- SQL sampling from PostgreSQL stats views.
- Direct DB optional load check.
- Markdown/JSON report generation.
- `loadtest:doctor` readiness check for env parsing and k6 availability.
- Docs, release notes, follow-ups, phase log, original review, and handoff.

Non-goals:

- No Admin UI pages.
- No browser controls to run, stop, or schedule tests.
- No App-Service-hosted load generation.
- No Azure Container Apps, Container Instances, Azure Load Testing, queues,
  functions, Blob Storage, or new cloud dependencies.
- No required Azure metric sampling.
- No huge raw k6 event streams in PostgreSQL.
- No production app runtime behavior changes.
